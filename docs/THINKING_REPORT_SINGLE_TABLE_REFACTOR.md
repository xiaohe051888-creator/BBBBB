# 深度思考与逻辑评估报告：系统单桌化与上传/结束本靴工作流重构

## 1. 需求解析
根据您的最新要求，系统需要进行两项重大重构：
1. **彻底移除多桌台（table_id）逻辑**：整个系统只保留一个桌子，不再需要桌台选择。
2. **重构仪表盘顶部操作流**：
   - 顶部增加“结束本靴”按钮和“上传数据”按钮。
   - 点击“结束本靴”：自动结束当前靴，并跳转到上传数据页面（此时上传的数据将被视为**新靴**）。
   - 点击“上传数据”：停留在当前靴，上传的数据只**覆盖/追加**当前靴的数据。
   - 无论哪种方式上传，确认上传后，都会强制结束当前正在进行的工作流（如进行中的AI分析、等待开奖等），并基于新数据重新开始工作。

---

## 2. 移除多桌台（table_id）的深度评估
### 2.1 当前系统的强依赖
经过对代码库的深度排查，目前 `table_id` 是系统的核心基石，贯穿了所有层级：
*   **数据库**：所有 8 张核心表（如 `GameRecord`, `BetRecord`, `SystemState`, `MistakeBook` 等）都使用 `table_id` 作为联合主键或过滤条件。
*   **后端 API**：所有的请求入参（Schemas）和查询参数都强制要求 `table_id`。
*   **WebSocket**：目前的 WS 房间是按 `/{table_id}` 划分的。
*   **前端状态与路由**：React Query 的缓存 Key 全部绑定了 `tableId`，且 React Router 强依赖 `/dashboard/:tableId` 的路由格式。

### 2.2 移除的风险与工作量
*   **工作量极大且极易出错**：移除 `table_id` 意味着要修改几乎所有的后端查询语句、数据库迁移（Schema 变更）、前端的所有 API 调用、所有的 React Query 缓存键以及整个 React Router 结构。
*   **破坏性变更**：这会导致现有的数据库文件完全失效，必须删库重建。
*   **建议方案（优雅降级）**：
    不要在代码物理层面上大动干戈地删除 `table_id` 字段，而是**在前端路由和入口处将其“隐藏”并写死**（例如全局默认使用 `table_id = "T1"` 或 `"DEFAULT"`）。
    *   去掉前端的“选择桌台”页面。
    *   将 `/dashboard/:tableId` 路由简化为 `/dashboard`。
    *   前端在调用所有 API 和 WS 时，底层自动带上写死的 `table_id`。
    *   *这样做不仅能 100% 满足您“用户视角只有一个桌子”的需求，还能避免对稳定后端底层的破坏性手术。*

## 3. “结束本靴”与“上传数据”逻辑的深度思考
### 3.1 现有的系统逻辑
目前系统的设计是：**只要上传数据，就默认是硬重置（清空当前桌的所有历史，开启新靴）**。后端 `upload_games` 接口会强制把内存状态清空，并自动计算新的 `boot_number`。现有的 `end_boot` 接口则是用于触发 AI 深度学习，并在完成后将 `boot_number + 1`。

### 3.2 您的新逻辑带来的冲突与风险
您的新逻辑将上传行为拆分为了两种语义：**“覆盖当前靴”** 和 **“开启新靴”**。

#### 风险一：靴号（boot_number）的状态错乱
*   **当前靴覆盖逻辑**：如果您点击“上传数据”只是覆盖当前靴，这就意味着后端在接收到数据时，**不能**再去清空数据库中当前靴的历史数据，也不能去 `boot_number + 1`。它必须精准地根据当前活跃的 `boot_number` 执行 `DELETE + INSERT` 或 `UPSERT` 操作。
*   **新靴逻辑**：如果点击“结束本靴”，跳转到上传页后上传的数据是新靴，那么前端必须在跳转时携带一个状态（例如 `isNewBoot=true`）。后端在接收到这个标志时，才执行 `boot_number + 1` 并开启新纪元。

#### 风险二：工作流（Workflow）的强行截断
您提到：“不管用什么方式上传，确认后整个工作流就结束前面的工作，重新开始”。
*   **当前逻辑的盲区**：目前如果系统正处于“等待开奖”或“AI分析中”的状态，强制上传会直接破坏内存中的状态机。例如，如果有已下注但未开奖的记录（Pending Bet），强制覆盖数据会导致该注单永远无法结算，资金出现幽灵扣款或错乱。
*   **解决对策**：后端 `upload_games` 接口必须增加一层**强力清场逻辑**：
    1. 强制将所有未结算的注单（Pending Bets）标记为“作废”或“退回”。
    2. 强制中断正在执行的异步 AI 分析任务（Cancel Task）。
    3. 清理五路图引擎缓存。
    4. 然后再写入新数据，并重新触发新一轮的 AI 分析。

#### 风险三：错题本与深度学习的断层
*   目前系统的 `end_boot`（结束本靴）除了增加靴号，更重要的职责是**触发本靴的错题本深度学习**。
*   如果用户点击“结束本靴”后立刻跳转去上传新数据，此时后端的“深度学习”异步任务可能还没跑完（需要耗时几十秒）。如果新数据立刻涌入，会污染深度学习正在读取的上下文数据。
*   **解决对策**：点击“结束本靴”时，前端必须弹出一个过渡层，等待后端返回“深度学习完成”的 WebSocket 信号后，才允许用户在上传页点击“确认上传”；或者，前端在跳转到上传页时，将上传按钮置灰（显示“正在结算与学习中...”），学习完成后才放开限制。

## 4. 总结与行动建议
您的逻辑构想非常符合单机单用户的极简操作体验，但在实现上需要进行精细的状态机控制。

**我为您总结的行动路线图如下：**
1.  **全局隐式单桌化**：保留后端 `table_id`，前端移除选择页，路由抹除 `:tableId`，底层 API 请求统一写死 `table_id="DEFAULT"`。
2.  **前端顶部栏重构**：在 Dashboard 顶部 Header 增加【结束本靴】和【上传数据（覆盖本靴）】按钮。
3.  **后端 Upload API 升级**：修改 `upload_games` 接口，增加 `is_new_boot: bool` 参数。
    *   若为 `false`，则清除当前靴记录并写入新记录，保持 `boot_number` 不变。
    *   若为 `true`，则 `boot_number + 1`。
    *   无论哪种情况，都执行“强力清场逻辑”（退回未结算注单、终止运行中的 AI）。
4.  **深度学习阻塞机制**：点击“结束本靴”后，前端进入上传页并锁定上传按钮，直到接收到上一靴深度学习完成的信号。

这个逻辑闭环是严密且可行的，只要处理好异步任务的截断与等待，就不会出现数据丢失或状态错乱。您可以确认这个路线图是否符合您的预期，如果确认，我们可以开始逐步实施。
