# QA 收尾优化（文案一致性 + /ws HTTP 明确拒绝）设计

日期：2026-05-02  
目标：在不改变主流程与安全边界的前提下，修复狗粮测试发现的 3 个小问题，提升一致性与可诊断性。

## 问题与目标

### 1) 登录按钮文案不一致

- 现状：登录弹窗按钮显示“登录进入管理面板”，但登录成功后跳转到 `/mode`（模式选择页）
- 目标：按钮文案与实际行为一致

### 2) 修改密码弹窗标题不一致

- 现状：已取消“强制修改默认密码”，但弹窗标题仍为“修改默认密码”
- 目标：统一为“修改密码”，避免误导

### 3) `/ws` 普通 HTTP GET 返回 200 易误判

- 现状：对 `/ws` 进行普通 HTTP GET 可能返回 200
- 目标：当不是 WebSocket Upgrade 请求时，明确返回 426（Upgrade Required）+ 简短提示信息；WebSocket 握手不受影响

## 方案（推荐）

### 前端

1. `LoginModal` 主按钮文案改为“登录进入系统”
2. `AdminPage` 修改密码弹窗标题改为“修改密码”

### 后端

在现有 WebSocket 路由文件中增加一个 HTTP GET `/ws`：
- 如果请求不是 WebSocket Upgrade：返回 426，并提示“请使用 WebSocket Upgrade 连接 /ws”
- WebSocket 路由仍由 `@router.websocket("/ws")` 处理，不受影响

## 验收标准

- 登录弹窗主按钮文案与实际跳转一致
- 管理页修改密码弹窗标题为“修改密码”
- `curl http://<host>/ws` 返回 426
- 原有 WebSocket token 校验仍然生效（无 token 连接失败）
