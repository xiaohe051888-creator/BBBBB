# Analysis Detail Readability And History Reset Design

**Date:** 2026-05-10

**Goal:** 修复 `推理详情` 在移动端和低对比场景下“层级乱、颜色淡、字小、信息密”的可读性问题；同时修复 `reset_current_boot` 后历史开奖记录里的 `预测 / 正确 / 盈亏` 被清成 `- / 0.00` 的问题，保证历史展示在重置当前靴后仍然可信。

**Architecture:** 前端把详情抽屉重构为“结论优先、证据分组、技术信息后置”的移动端优先阅读结构；后端在 `reset_current_boot` 覆盖当前靴前，先抓取当前靴已结算历史局的展示快照，待新开奖记录写回后再按局号恢复这些历史展示字段，避免因删除 `BetRecord` 导致 `GameRecord` 展示字段整体丢失。

**Tech Stack:** React, TypeScript, Ant Design Drawer/Collapse, FastAPI, SQLAlchemy, pytest, Vitest

---

## 1. Problem Statement

### 1.1 详情抽屉的真实问题

当前 `AnalysisDetailDrawer` 已经有正确的信息结构，但真实使用时仍有以下问题：

- 信息层级不够明显，用户进入后不知道先看“结论”还是“五条路”
- 正文颜色偏淡，和背景对比不足，长时间阅读费力
- 字号与留白偏紧，移动端上尤其容易显得拥挤
- “技术说明”虽然已折叠，但上方正文仍然像连续说明文，不像分段卡片
- 每条路的说明缺少视觉锚点，用户很难快速扫读“这条路偏庄/偏闲/支持强弱”

### 1.2 历史记录被清空的真实问题

当前 `reset_current_boot` 逻辑会删除当前靴全部 `BetRecord`。虽然这是为了重置工作流状态，但前端开奖记录展示依赖 `GameRecord` 上的这些历史字段：

- `predict_direction`
- `predict_correct`
- `settlement_status`
- `profit_loss`
- `balance_after`

这些字段在历史上本来来自结算流程写回；一旦覆盖上传仅保留新的开奖结果，而删除旧下注记录与历史展示值，前端就会看到：

- `预测 = -`
- `正确 = -`
- `盈亏 = 0.00`

这不是用户理解上的“新数据覆盖”，而是展示信息被错误抹掉。

## 2. UX Design

### 2.1 总体目标

详情页改成一眼能扫读的“先结论、再理由、后细节”结构，优先满足手机端单手阅读。

### 2.2 新的阅读顺序

抽屉内容固定按以下顺序出现：

1. `本局结论`
2. `最终为什么押这个方向`
3. `五条路怎么看`
4. `来源说明`
5. `技术说明（折叠）`

其中：

- `本局结论` 负责回答“这局押什么”
- `最终为什么押这个方向` 负责回答“为什么最后是这个结论”
- `五条路怎么看` 负责回答“每条路各自怎么支持或提醒风险”
- `来源说明` 负责解释“这是单AI还是规则兜底”
- `技术说明` 只给排障，不参与主阅读流

### 2.3 视觉与排版调整

详情抽屉采用移动端优先样式：

- 抽屉宽高保持现有交互模式，但正文区域改成更明显的卡片分区
- 主结论区域增加方向、把握度、来源的高对比度展示
- 正文字号提升，正文行高提升，段落间距增大
- 各 section 使用白底/浅灰底卡片或高对比边框分隔，避免整页文本连成一片
- 关键标题使用更强层级，次级辅助说明用较弱颜色，但不得淡到影响阅读
- 每条路卡片都固定展示：
  - 路名
  - 当前倾向
  - 支持强弱
  - 一句趋势标签
  - 一段小白解释

### 2.4 五条路卡片规则

每条路都需要满足“扫一眼能知道立场”的要求：

- 标题行显示路名
- 右侧显示 `庄 / 闲 / 中性` 与 `强 / 中 / 弱`
- 趋势标签单独一行显示，作为短结论
- 小白解释单独成段，不和标题、标签混排

移动端下卡片必须单列堆叠，避免横向拥挤；桌面端也保持单列阅读，优先可读性而非信息压缩。

### 2.5 技术说明的位置

技术说明继续保留折叠区，但明确降级：

- 默认折叠
- 放在页面最后
- 文案颜色不应强于主结论区
- 仅在存在 `technical_diagnostic.message` 时展示

## 3. Data Preservation Design

### 3.1 需要保留的历史展示字段

`reset_current_boot` 时，必须保留当前靴内“已经开奖并已经结算”的历史展示结果：

- `predict_direction`
- `predict_correct`
- `settlement_status`
- `profit_loss`
- `balance_after`
- `prediction_mode`

这些字段属于历史展示事实，不应因为重新上传同一靴的开奖结果而消失。

### 3.2 不应保留的运行态字段

以下字段必须继续清空，以避免把旧工作流状态带到新流程：

- 内存会话中的 `pending_bet_*`
- 内存会话中的 `predict_*`
- 当前分析缓存与 `analysis_outcome`
- 当前待开奖注单
- 当前靴未完成的后台任务态

因此，这次修复不是“不删下注记录”，而是“在删除前提下保留历史展示快照”。

### 3.3 推荐实现

在 `backend/app/services/game/upload.py` 的 `reset_current_boot` 分支内：

1. 删除当前靴数据之前，按当前 `boot_number` 查询已有 `GameRecord`
2. 将已存在记录中可展示的历史字段按 `game_number` 采集为快照
3. 继续执行当前的清理逻辑，删除 `GameRecord / BetRecord / MistakeBook / RoadMap / AIMemory`
4. 重新写入上传的开奖记录
5. 对每一条重新写入后的 `GameRecord`：
   - 如果该 `game_number` 在历史快照中存在，则恢复展示字段
   - 仅恢复历史展示字段，不恢复运行态或 pending 态

### 3.4 恢复规则

恢复时遵循这些规则：

- 只按 `game_number` 恢复，不依赖旧 `id`
- 如果上传的新数据局号不存在于历史快照，则保持空展示字段
- 若历史快照中的某局原本没有结算值，则不强行写入默认值
- `profit_loss` 不得被强制置为 `0.00`，只有真实为零时才保留零

### 3.5 为什么不用联表临时修前端

仅靠前端或 `/games` 动态联表 `BetRecord` 不能解决根因，因为 `reset_current_boot` 目前会删除整靴下注记录。只改查询层仍然会得到空值，必须在重置流程里保护历史展示数据。

## 4. Component Design

### 4.1 `AnalysisDetailDrawer`

保留现有组件名，但内部改为以下结构：

- `SummaryCard`
  - 显示方向、来源、把握度、一句话总结
- `FinalReasonCard`
  - 单独显示 `最终为什么押这个方向`
- `RoadExplanationCards`
  - 顺序渲染五条路卡片
- `SourceReasonCard`
  - 显示来源说明
- `TechnicalDiagnosticCollapse`
  - 显示技术说明

不要求拆成五个独立文件，但实现上应具备这五块的清晰边界，避免所有内容混在一个连续 JSX 中。

### 4.2 `GameRecord` 历史恢复

上传重置逻辑增加一个内部辅助函数，例如：

- `_capture_game_history_snapshot(...)`
- `_restore_game_history_snapshot(...)`

职责明确：

- 一个负责在删除前提取旧值
- 一个负责在新记录写入后按局号恢复旧值

这样可以避免把上传主流程写成一大段难维护的内联逻辑。

## 5. Testing

### 5.1 前端测试

补充或更新 `frontend/src/components/dashboard/AnalysisDetailDrawer.test.tsx`，至少覆盖：

- 结论区、最终原因、五条路、来源说明的顺序存在
- 技术说明默认仍为折叠区域
- 关键标题和正文文案都能渲染
- 新增的高层级文本块在 DOM 中分区存在

如果现有测试方式不足以表达结构变化，可以增加针对 section 标题与关键文案的断言。

### 5.2 后端测试

新增或更新上传相关测试，至少覆盖：

- 先构造一个有历史 `predict_direction / predict_correct / profit_loss` 的当前靴
- 执行 `reset_current_boot`
- 再读取同靴 `GameRecord`
- 断言历史局的展示字段仍然保留
- 断言 pending/分析中的运行态没有被错误恢复

### 5.3 手工复测

实现完成后需要再次线上验证：

1. 用真实账号进入 `dashboard`
2. 打开 `查看详细原因`
3. 在手机宽度或窄窗口下确认详情可读性显著提升
4. 执行一次 `reset_current_boot`
5. 确认历史开奖记录不再大面积出现 `- / 0.00`

---

## 6. Scope Boundaries

本次设计明确不做以下内容：

- 不重做首页 `AnalysisPanel` 主结果卡的大结构
- 不更换 `Drawer` 为全新路由页面
- 不改下注、开奖、规则兜底的核心业务规则
- 不把所有历史展示改成实时联表动态计算

本次只修：

- 详情抽屉的可读性
- `reset_current_boot` 导致的历史展示丢失

---

## 7. Success Criteria

满足以下条件视为完成：

- 用户在手机端打开详情抽屉后，能快速看清本局结论、最终原因、每条路说明
- 详情页不再出现“整页都是密集灰字”的观感
- 执行 `reset_current_boot` 后，历史局的 `预测 / 正确 / 盈亏` 不再错误回退为 `- / 0.00`
- 线上真实流程复测中，这两个修复都能被实际观察到
