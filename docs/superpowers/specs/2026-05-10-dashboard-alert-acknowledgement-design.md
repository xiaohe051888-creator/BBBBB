# 首页严重告警确认隐藏 设计

日期：2026-05-10  
范围：仅管理员首页告警条，不影响记录页  
目标：管理员点一次“确认”后，首页严重告警条立即消失；后续只有出现新的高优先级告警时才重新出现；记录页保留全部日志

## 背景

当前首页顶部的 `严重告警` 条只要最近 24 小时内存在 `P1` 日志就会持续显示。  
这会带来两个问题：

1. 管理员已经看过并理解当前告警，但首页仍长期占位，没有“已确认”的交互闭环。
2. 记录页和首页承担了同一批信息的双重职责，首页无法表达“我知道了，但请保留审计记录”。

用户明确要求的目标是：

- 点一次确认后，首页消失
- 记录页保留
- 如果后续出现新的高优先级告警，首页应自动再次提示

## 目标

1. 为管理员首页告警条增加“确认”能力
2. 确认后仅隐藏首页告警条，不删除、不修改日志记录
3. 刷新页面、重新登录、切换设备后，已确认状态仍然生效
4. 只要出现新的 `P1` 告警，首页自动恢复显示

## 非目标

- 不做每条日志逐条确认
- 不做“已确认告警”单独列表
- 不删除、不软删除、不修改 `SystemLog`
- 不改变记录页、导出、日志详情页的展示口径

## 方案对比

### 方案一：仅前端本地隐藏

- 做法：点击确认后把状态写进 `localStorage`
- 优点：改动最小
- 缺点：换设备、换浏览器、清缓存后失效，无法表达管理员账号级确认

### 方案二：按管理员账号记录“已确认到的最新告警”

- 做法：后端保存管理员已确认的最新 `P1` 告警标记，前端只显示未确认的新告警
- 优点：跨刷新、跨设备一致；不会污染日志数据；能自然支持“出现新告警后重新显示”
- 缺点：需要一条很小的后端持久化字段和一个确认接口

### 方案三：给每条日志增加确认状态

- 做法：对每条 `SystemLog` 单独标记是否确认
- 优点：粒度最细
- 缺点：对当前需求过重，且会把首页提醒能力膨胀成日志工单系统

## 选型

采用方案二：按管理员账号记录“已确认到的最新告警”。

原因：

- 它正好满足“首页消失、记录页保留、出现新告警后再次出现”这三个核心目标
- 它与当前管理员登录体系天然兼容
- 它不会把首页提醒逻辑和日志审计数据耦合到一起

## 数据设计

在 `AdminUser` 上新增一个可空字段：

- `acknowledged_alert_log_id: int | null`

语义：

- 表示该管理员已经确认到的最新严重告警日志编号
- 该值为空时，表示从未确认过首页严重告警
- 该值不回写 `SystemLog`，只作为首页提醒视角的个人确认进度

选择 `log_id` 而不是时间戳的原因：

- `SystemLog.id` 是单调递增的主键，更适合表达“我已经确认到哪一条”
- 比单纯比较时间更稳定，避免同秒日志排序歧义

## 后端设计

### 1. 扩展 `GET /api/admin/maintenance/alerts`

当前接口只返回：

- `count`
- `data`

扩展后返回：

- `count`：最近 24 小时内严重告警总数
- `data`：严重告警列表
- `latest_alert_log_id`：当前结果集内最新一条严重告警的日志编号，没有则为 `null`
- `acknowledged_alert_log_id`：当前管理员已确认到的日志编号，没有则为 `null`
- `unacknowledged_count`：当前仍未确认的严重告警数

计算规则：

- 最近 24 小时内符合条件的 `P1` 日志仍按原逻辑查询
- `latest_alert_log_id` 取当前结果集第一条日志的 `id`
- `unacknowledged_count` 统计 `id > acknowledged_alert_log_id` 的日志数
- 当没有任何严重告警时：
  - `count = 0`
  - `latest_alert_log_id = null`
  - `unacknowledged_count = 0`

### 2. 新增 `POST /api/admin/maintenance/alerts/acknowledge`

请求体：

- `latest_alert_log_id: number`

行为：

1. 校验管理员身份
2. 查询当前管理员
3. 将 `acknowledged_alert_log_id` 更新为：
   - `max(当前值, latest_alert_log_id)`
4. 返回更新后的确认结果

返回体：

- `acknowledged_alert_log_id`
- `latest_alert_log_id`
- `unacknowledged_count`

接口约束：

- 如果当前没有严重告警，接口允许幂等返回，不报错
- 如果前端传入的 `latest_alert_log_id` 小于数据库中当前已确认值，不回退
- 如果传入的 `latest_alert_log_id` 不是最近窗口内的真实严重告警编号，后端应按当前窗口真实最新值兜底

### 3. 数据迁移

新增 Alembic migration，为 `admin_users` 表增加：

- `acknowledged_alert_log_id INTEGER NULL`

此字段默认空值，不影响现有管理员登录流程

## 前端设计

### 1. 扩展首页告警条显示逻辑

`AdminAlertsBar` 改为以 `unacknowledged_count` 为首页显示依据：

- 未登录：admin 不显示
- `count == 0`：不显示
- `count > 0` 且 `unacknowledged_count == 0`：不显示
- `unacknowledged_count > 0`：显示

这样首页只展示“有新的、还没确认过的严重告警”。

### 2. 新增“确认”按钮

按钮文案：

- `确认`

交互：

1. 管理员点击后，请求 `POST /alerts/acknowledge`
2. 成功后立即隐藏首页告警条
3. 不跳转页面，不清空日志，不影响记录页
4. 如果稍后轮询发现有新的未确认严重告警，再自动重新显示

现有按钮调整：

- `刷新` 保留
- `展开 / 收起` 保留
- `查看全部` 保留
- `确认` 新增为首页主动作之一

### 3. 文案原则

首页确认按钮的语义是：

- “我已知晓当前严重告警”

而不是：

- “问题已解决”
- “删除这条告警”
- “忽略所有未来告警”

因此首页隐藏只是提醒层收口，不是运维状态清零

## 数据流

1. 管理员登录 Dashboard
2. 前端轮询 `GET /api/admin/maintenance/alerts`
3. 后端返回最近窗口内严重告警 + 当前管理员确认进度
4. 前端根据 `unacknowledged_count` 决定是否显示首页告警条
5. 管理员点击 `确认`
6. 前端调用 `POST /api/admin/maintenance/alerts/acknowledge`
7. 后端更新 `AdminUser.acknowledged_alert_log_id`
8. 前端更新本地状态，首页立即隐藏
9. 后续若出现新的 `P1` 日志，轮询返回 `unacknowledged_count > 0`，首页自动重新显示

## 错误处理

- 获取告警失败：维持当前前端容错策略，不展示错误弹窗轰炸首页
- 确认失败：提示“确认失败，请稍后重试”，但不隐藏告警条
- 后端无管理员记录：返回鉴权/身份错误，与现有 admin-only 接口口径一致
- 新旧请求乱序：以后端返回的最新确认值为准，前端不自行推断更深状态

## 测试策略

### 后端

新增测试覆盖：

1. `GET /alerts` 返回 `latest_alert_log_id / acknowledged_alert_log_id / unacknowledged_count`
2. `POST /alerts/acknowledge` 成功写入管理员确认标记
3. 已确认后再次拉取 `GET /alerts`，`unacknowledged_count` 正确变为 0
4. 新增一条更新的 `P1` 日志后，`unacknowledged_count` 重新大于 0
5. 确认接口对旧编号请求保持幂等，不回退确认进度

### 前端

新增测试覆盖：

1. 有未确认告警时显示首页告警条
2. 点击 `确认` 后首页立即隐藏
3. 重新拉取仍全部已确认时，首页保持隐藏
4. 拉到新的未确认告警后，首页重新显示
5. `查看全部` 仍跳到日志页，记录页数据不受影响

## 验收标准

1. 管理员点击一次 `确认` 后，首页严重告警条立即消失
2. 刷新页面、重新登录后，在没有新严重告警的情况下，首页仍保持隐藏
3. 记录页依然能看到原始日志和详情
4. 出现新的 `P1` 严重告警后，首页自动再次显示告警条
5. 整个过程不删除、不修改 `SystemLog` 记录内容
