# 系统全面完善（稳定性 + 可观测性 + Postgres 化）设计稿

日期：2026-04-30  
范围：全栈（后端 FastAPI + 前端 React）、部署（Docker Compose + Render）  
目标：在保持现有业务语义与交互不变的前提下，将系统提升到“可长期运行 + 可上线部署 + 关键功能补齐”的交付标准

## 背景与现状

系统当前形态是“前端 React + 后端 FastAPI + 内存会话 ManualSession + 数据库存储”的全流程托管工具，已具备：

- 上传路单（覆盖本靴 / 开启新靴）→ 触发分析 → 自动下注 → 开奖 → 结算 → 触发下一局分析的闭环
- 管理员：模式切换、学习启动、接口配置、数据查看
- WebSocket：推送系统状态与事件

近期已修复/增强（摘要）：

- 上传链路默认不触发学习；学习改为管理员手动触发，且支持全库学习
- “连续20局”类提示去重，避免刷屏
- 关键错误码（如 illegal_state）前端兜底中文化
- 后端分析任务增加 stale_boot 防串写保护
- 开奖接口增加幂等能力（已开奖可重复请求，不重复写入）

仍存在的核心不足集中在：

- 稳定性：状态机/并发/后台任务边界尚未体系化，存在回归风险
- 可观测性：错误码、日志、事件口径不统一，排障成本高
- 数据层：当前仍以 SQLite 为主，存在连接生命周期与长期运行风险；云部署也更适合 Postgres
- 功能补齐：日志页等存在占位组件；部分管理配置展示含技术字段，需进一步统一与增强

## 总体原则

1. 不改变现有业务语义与主要交互（除非属于“明显缺陷修复/提示优化”）。
2. 关键链路必须可回归：上传、分析、下注、开奖、结算、学习启动、管理员登录。
3. 每个阶段都要可验收：阶段结束时测试通过、构建通过、并给出可验证的操作清单。
4. 数据层以 Postgres 为长期运行目标；本地仍保留最小可用开发路径（可用 SQLite 或本地 Postgres）。

## 方案对比与选择

### 方案 A：只做补丁式修复

- 优点：改动小、推进快
- 缺点：状态机/并发/错误口径仍分散，越改越难验证，长期成本更高

### 方案 B：分阶段系统化加固（推荐）

- 优点：每阶段可验收，逐步收敛状态机与错误口径，能持续验证不回归
- 缺点：需要引入明确的抽象与测试结构，初期工作量更大

### 方案 C：推倒重写

- 优点：可以一次性重塑架构
- 缺点：风险极高，容易破坏既有业务语义，不适合当前目标

选择：方案 B。

## 分阶段交付设计

### 第一阶段：系统理解与风险清单固化

输出物：

- “系统关键链路”与“状态机”文档：明确每个状态下允许的操作与返回
- 风险清单：并发/幂等/错误码/数据一致性/性能热点
- 回归清单：按页面与接口列出手工验证路径

验收：

- 文档可被复述与执行（按清单操作能复现每条链路）

### 第二阶段：后端稳定性加固（状态机 + 并发 + 幂等）

核心设计：

- 统一错误结构：服务层返回统一结构（success/error/message），路由层统一映射 HTTP 状态码（400/409/500）
- 状态机集中化：将“允许的状态转移”和“禁止原因”集中到单点（避免散落字符串判断）
- 幂等策略：
  - 开奖：已开奖重复请求返回 success + already_revealed
  - 上传：覆盖本靴/开启新靴必须是可重复的“最终一致结果”（相同输入不产生新副作用）
  - 学习触发：重复触发要么拒绝要么复用任务（避免重复跑）
- 后台任务治理：
  - 所有异步任务带 boot_number 上下文，回写前强校验（已做 stale_boot，但要覆盖更多入口）
  - 统一任务注册/清理（避免 GC、避免重复任务堆积）

回归测试（示例）：

- upload / reveal / analysis / learning 的核心路径与关键非法状态
- stale_boot 丢弃、重复开奖幂等、错误码映射到中文 detail

验收：

- 后端 tests 全绿
- 关键接口在“重复点击/重复请求/刷新后重试”下行为可预测

### 第三阶段：数据层迁移到 Postgres（长期运行目标）

核心设计：

- 以环境变量 DATABASE_URL 驱动数据库选择：生产默认 Postgres
- 数据模型迁移：
  - 保持现有 SQLAlchemy 模型不变（必要时补齐字段类型兼容）
  - 引入迁移/初始化策略（如 Alembic 或“启动时建表+版本标记”）
- 数据迁移策略（SQLite → Postgres）：
  - 提供一次性导出导入脚本（导出为 JSON/CSV 或直接 SQLAlchemy 读取写入）
  - 明确哪些表需要迁移：GameRecord、BetRecord、MistakeBook、RoadMap、AIMemory、SystemLog、ModelVersion 等
- 连接生命周期修复：
  - 统一使用 async_session 且在应用退出时优雅关闭
  - 清理 ResourceWarning 的根因（连接未关闭/事件循环销毁前未 close）

验收：

- docker-compose.prod.yml 可无改动起 Postgres（或补齐 compose 里的 postgres service）
- Render 部署路径可用（现有 render.yaml 已提供 Postgres）
- 同一套功能在 Postgres 下跑通回归清单

### 第四阶段：可观测性与“全中文可见文本”收敛

核心设计：

- 前端统一错误呈现：
  - 后端 detail 若为 error code（例如 illegal_state），前端统一映射为中文（已部分实现，补齐覆盖）
  - 关键错误给出“下一步建议”（刷新/重试/去管理员页等）
- 日志与事件口径：
  - 事件类型、结果、优先级在前端统一中文展示
  - 导出功能对用户只展示“表格/数据”，不暴露英文缩写

验收：

- 用户界面除“AI”外不出现英文错误码/英文提示
- 排障路径清晰：任意异常可从“系统状态面板 + 实盘日志”定位

### 第五阶段：功能补齐（重点：日志页）

目标：

- 去掉占位组件，补齐可用功能：
  - 过滤栏（分类/优先级/关键词/时间范围）
  - 详情弹窗（完整字段、复制、跳转到关联局号/靴号）
  - 统计区（按分类/优先级聚合）

验收：

- 日志页可独立用于排障（不需要看后端日志文件也能定位主要问题）

## 非目标（明确不做）

- 不重写前端为新框架、不引入重型 UI 体系迁移
- 不改变核心策略算法与胜率逻辑（除非是明显 bug 修复）
- 不在本阶段引入复杂的权限体系（仅保持现有管理员 token 逻辑并加固）

## 风险与缓解

- Postgres 迁移风险：通过“并行验证（本地可选 SQLite/PG）+ 回归清单 + 可回滚脚本”控制
- 状态机改造风险：先用测试锁定行为，再重构集中化，避免一次性大改
- 文案全中文风险：仅改用户可见文本，不动协议字段与内部变量

## 验收总清单（最终）

- 前端：lint/build 通过；用户可见提示全中文（保留“AI”）；关键页面手工回归无异常
- 后端：tests 全绿；关键接口幂等；后台任务无串写；错误码口径统一
- 部署：docker compose 与 Render 路径可用；Postgres 为主数据库；提供备份/迁移说明
