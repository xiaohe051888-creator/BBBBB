# DeepSeek V4 Pro 单AI正式调用修复设计

## 背景

当前单AI模式已经固定在 DeepSeek 路线，但线上仍出现“配置测试成功、正式分析预测持续失败或超时”的问题。现有代码表明，测试链路与正式分析链路并不完全一致：

- 测试接口只发送极轻量的 `"Hello"` 探活请求，并以 10 秒超时校验接口可达。
- 正式分析会携带历史局、五路数据、错题上下文、路特征和提示词模板，负载远高于测试。
- 正式分析当前默认请求超时较短，外层总分析超时也偏紧，且与内部重试预算不匹配。
- 测试链路会对兼容地址进行补全与规范化，但正式分析链路仍有单独拼接 `chat/completions` 的逻辑，存在“测试通过、正式地址跑偏”的风险。

用户已明确约束：

- 单AI模式固定使用 DeepSeek `V4 Pro`
- 单AI模式固定开启深度思考进行推理
- 必要时允许完全重构单AI正式调用与提示词

## 问题定义

本次要解决的不是“单AI能否探活”，而是“单AI在 DeepSeek V4 Pro 且深度思考开启时，能否稳定完成正式分析预测并进入下注链路”。

成功标准：

1. 单AI正式分析不再因为测试链路和正式链路不一致而跑偏。
2. 单AI正式分析在 DeepSeek V4 Pro + `thinking=enabled` 下具备合理的超时预算。
3. 管理页不再允许把单AI配置成与设计目标不一致的其它模型。
4. 单AI测试结果要更接近实盘，而不是只证明“接口能回 200”。
5. 单AI失败时的诊断信息能直接看出是地址、超时、上游报错还是返回格式问题。

## 方案对比

### 方案 A：只调大超时

优点：

- 改动最小
- 可快速验证是否纯粹因为 45 秒总超时过紧

缺点：

- 无法解决测试链路与正式链路地址处理不一致的问题
- 无法避免配置页把单AI切成其它模型
- 即便超时放宽，也可能只是把失败变慢

### 方案 B：关闭深度思考换稳定性

优点：

- 最可能立刻降低耗时
- 正式链路成功率可能快速上升

缺点：

- 违背用户要求
- 会改变单AI的分析特征，不符合本次目标

### 方案 C：全链路修复

内容：

- 统一测试链路与正式链路的 DeepSeek 地址规范化
- 固定单AI模型为 `deepseek-v4-pro`
- 固定 `thinking=enabled`
- 重新设计单AI正式请求超时、总分析超时、重试预算
- 升级单AI测试为“轻量探活 + 正式结构模拟测试”
- 补充明确的运行诊断与回归测试

结论：

采用方案 C。

## 设计

### 1. 单AI运行时配置固定化

单AI模式的运行时配置收敛为 DeepSeek 专用：

- `provider` 固定为 `deepseek`
- `model` 固定为 `deepseek-v4-pro`
- `thinking` 固定为 `enabled`

管理页中的单AI配置不再允许自由切换模型。页面上仍保留：

- 访问密钥
- DeepSeek 兼容接口地址
- 单AI预测提示词模板
- 单AI等待开奖阶段策略提示词模板

这样可以避免：

- 配置页保存为其它模型后，正式分析与产品设计偏离
- 测试通过的是某个轻模型，实盘跑的是另一模型

### 2. DeepSeek 地址构造统一

新增统一的单AI上游地址构造逻辑，由测试接口和正式分析共同复用。

规则：

- 如果未填写地址，DeepSeek 默认使用 `https://api.deepseek.com`
- 统一在地址构造器中补全兼容路径
- 不允许测试接口和正式分析分别维护不同的拼接规则

统一目标：

- 同一份配置，在测试接口与正式分析中命中的最终 URL 一致
- 诊断信息中可直接看到规范化后的目标地址

### 3. 正式分析超时与重试策略重构

单AI正式分析在 DeepSeek `V4 Pro + thinking=enabled` 下天然更慢，因此需要把“单次请求超时”和“整次分析超时”拆开设计。

设计原则：

- 正式分析超时预算按 DeepSeek 深度思考模型的实际耗时设计
- 外层总分析超时必须覆盖内层单次请求和有限重试
- 避免当前这种“内部还在重试，外层 45 秒先把任务砍掉”的冲突

本次设计：

- 为单AI正式分析单独引入专用超时参数，不与通用轻量 API 测试混用
- 单次正式请求使用更宽的读取超时
- 正式分析保留有限重试，但重试次数收敛，避免无限消耗总预算
- 外层 `_run_followup_analysis()` 对单AI模式采用更长的总预算

### 4. 单AI测试升级

当前测试接口只能证明“接口能返回 200”，不能证明“正式分析能稳定工作”。

因此单AI测试升级为两段式：

1. 轻量连通测试
   - 校验密钥、基础地址、模型是否至少可达
2. 正式结构模拟测试
   - 使用小样本历史、简化五路数据和正式 JSON 输出约束
   - 走与正式分析相同的地址构造和单AI请求参数

最终展示给用户的“测试通过”，以正式结构模拟通过为准。这样测试通过更接近实盘可用。

### 5. 单AI提示词模板重构

单AI提示词允许完全重构，但目标不是“写得更长”，而是“让 DeepSeek V4 Pro 在深度思考开启时更稳定地产出严格 JSON”。

重构方向：

- 强化输出格式约束
- 避免无关寒暄与冗余解释
- 明确禁止 Markdown 包裹
- 提示模型先思考再只输出 JSON 结果
- 缩短非关键背景噪声，保留真正影响判断的结构化输入

同时保留占位符机制，继续支持：

- `{{BOOT_NUMBER}}`
- `{{GAME_NUMBER}}`
- `{{GAME_HISTORY}}`
- `{{ROAD_FEATURES}}`
- `{{ROAD_DATA}}`
- `{{MISTAKE_CONTEXT}}`
- `{{CONSECUTIVE_ERRORS}}`

### 6. 诊断信息增强

单AI正式分析失败时，诊断信息需能直接定位失败层级。新增或强化以下信息：

- 本次是否为 DeepSeek 单AI正式分析
- 规范化后的目标地址
- 实际模型名
- `thinking` 是否开启
- 失败类型：
  - 地址构造错误
  - 连接/读取超时
  - 上游状态码异常
  - 返回体缺失
  - JSON 解析失败

日志与前端展示仍保持用户可见内容中文化，但后台日志与诊断字段需要足够细，方便定位。

## 测试策略

本次至少补充以下回归：

1. 测试接口与正式分析复用同一地址构造器
2. 单AI配置保存后，运行时固定为 DeepSeek `V4 Pro`
3. 单AI正式调用固定附带 `thinking=enabled`
4. 单AI正式超时预算与外层总超时预算匹配
5. 单AI正式模拟测试通过时，模式可正常启用
6. 单AI正式调用超时或失败时，诊断信息包含地址、模型、思考状态和失败类型

## 风险与边界

- 本次不改变规则兜底继续下注机制；单AI失败后仍保持当前兜底能力。
- 本次重点修复单AI正式分析链路，不顺带重构三模型协作链路。
- 如果 DeepSeek 上游真实耗时显著高于预期，后续可能还需要继续微调超时预算，但本次必须先把链路一致性修正完成。
