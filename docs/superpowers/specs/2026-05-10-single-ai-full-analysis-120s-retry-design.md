# 单AI满血分析 120 秒窗口与手动重试设计

## 背景

当前单AI模式虽然已经完成了提示词收口、DeepSeek `V4 Pro + thinking=enabled` 固定化、解析真实性修复和部分前端状态收口，但系统目标仍然与用户期待存在偏差。

当前问题包括：

1. 单AI失败后仍存在降级到 `rule_fallback` 的路径
2. 智能分析面板没有把“为什么这次失败”讲清楚
3. 用户在 120 秒分析窗口失败后没有明确的人工介入入口
4. 系统过于偏向“流程继续”，而不是“坚持满血单AI分析质量”

用户这次给出的新硬约束非常明确：

- 只允许单AI模式做正式分析预测
- 必须是满血分析预测，不能缩水上下文、不能轻量化替代
- 只能在开奖录入完成后开始分析下一局
- 分析窗口允许到 `120` 秒
- 失败后不能降级
- 智能分析板块必须明确说明为什么失败
- 智能分析板块必须提供 `重新分析` 按钮
- 用户点击后，重新开始一轮新的 `120` 秒满血单AI分析

## 目标

本次设计目标有五个：

1. 把单AI模式重新定义成“满血单AI强约束模式”
2. 把单局正式分析窗口明确固定为开奖录入后的 `120` 秒
3. 去掉单AI失败后的自动降级路径，不再由规则兜底接管
4. 在智能分析面板中把失败原因讲清楚，让用户知道为什么没成功
5. 给用户提供可控的 `重新分析` 入口，手动重开一轮新的 `120` 秒分析

## 非目标

本次不处理以下事项：

- 不改三模型协作模式
- 不改普通规则模式
- 不追求“数学上绝对 100% 成功”
- 不在开奖前预测下一局
- 不引入本地默认庄闲结果
- 不在失败后偷偷走规则接管

## 核心原则

### 1. 满血分析原则

单AI模式下的每次正式分析都必须使用完整输入：

- 完整靴历史
- 完整五路原始数据
- 完整五路特征摘要
- 完整错题上下文
- 正式完整版提示词
- DeepSeek `V4 Pro + thinking=enabled`

不允许为了提速而：

- 只保留部分历史
- 只看部分五路
- 临时切成简化提示词
- 用轻量版分析代替正式分析

### 2. 开奖后启动原则

下一局预测只能在上一局开奖结果录入完成之后启动。

原因：

- 只有开奖录入完成后，系统才拥有真实完整的历史局面
- 在开奖前提前预测会污染输入完整性，损伤准确率
- 用户已明确指出：准确率是系统核心，不能为了提前启动而牺牲输入真实性

因此本次明确禁止：

- 在开奖前预启动下一局单AI预测
- 在历史未完整落盘前先做“预判型分析”

### 3. 禁止降级原则

单AI模式下，一旦进入正式分析：

- 不允许 `rule_fallback`
- 不允许默认结果补位
- 不允许把失败伪装成成功
- 不允许把“安全结果”包装成真实预测

也就是说：

- 成功，就展示真实单AI结果
- 失败，就明确展示失败态
- 重试，仍然是单AI重新做一轮满血分析

## 方案对比

### 方案 A：120 秒后自动规则兜底

优点：

- 用户大概率总能看到一个结果

缺点：

- 违背“不能降级”的核心要求
- 会让用户误以为看到的是单AI结果

结论：

不采用。

### 方案 B：120 秒后自动无限重试

优点：

- 用户不需要额外点击

缺点：

- 失败边界模糊
- 用户不知道当前到底是第几轮、为什么还没出结果
- 可能长期卡在“分析中”，体验反而更差

结论：

不采用。

### 方案 C：120 秒失败后进入明确失败态，由用户手动重开一轮新的 120 秒

优点：

- 状态边界清楚
- 失败原因可解释
- 既坚持满血单AI，又保留用户可控介入能力
- 不会把失败偷偷吞掉

结论：

采用方案 C。

## 设计

### 1. 单AI分析窗口重构

每次开奖录入完成后，系统进入一轮新的单AI正式分析窗口：

- 起点：开奖结果成功录入并写入系统
- 窗口：最多 `120` 秒
- 模式：满血单AI正式分析
- 输出目标：下一局 `庄 / 闲` 的真实单AI预测结果

在这一轮 `120` 秒窗口内，允许：

- 单AI内部请求重试
- 输出结构修复
- 结果校验

但所有补救动作都必须继续满足“满血单AI”原则，不能切换成降级路径。

### 2. 失败判定规则

当满足以下任一条件时，本轮进入失败态：

1. `120` 秒窗口耗尽，仍未形成有效结果
2. 上游模型连续返回无效结构，无法修复成合法结果
3. 上游服务在本轮窗口内持续不可用
4. 返回内容虽存在，但方向、字段或契约持续无效

失败态必须满足三条约束：

- 不写入成功 `analysis_outcome`
- 不显示“已完成研判”
- 不触发下注继续流转

### 3. 失败原因说明体系

智能分析板块的失败说明必须是中文、明确、可理解，不能只显示技术原文。

至少区分四类失败原因：

#### 3.1 分析超时

定义：

- 本轮满血分析在 `120` 秒内没有形成稳定结果

展示文案方向：

- `本轮满血分析在 120 秒内没有完成，因此当前还没有形成有效预测结果。`

#### 3.2 返回内容不完整

定义：

- 上游返回了内容，但缺少必须字段或结构不完整

展示文案方向：

- `这次分析已经返回内容，但结果不完整，系统无法把它当成有效预测结果。`

#### 3.3 返回方向无法识别

定义：

- 返回内容存在，但核心方向不是合法的 `庄 / 闲`

展示文案方向：

- `这次分析返回了内容，但没有形成可识别的庄闲方向，因此当前无法生成有效预测结果。`

#### 3.4 服务暂时不可用

定义：

- 上游服务超时、连接异常、限流或其它短时故障

展示文案方向：

- `这次满血分析在请求过程中遇到服务波动，因此当前还没有拿到稳定结果。`

#### 3.5 诊断补充原则

如果后端掌握更细诊断，应补充一段“补充说明”，但必须继续遵守：

- 不出现英文技术黑话
- 不把原始异常全文直接扔给用户
- 要先转换成“人能看懂的失败原因”

### 4. 智能分析面板状态设计

智能分析板块需要明确分成三类状态：

#### 4.1 分析中

用于 `120` 秒窗口内的进行态。

面板应可视化展示进度，而不是只有一句“分析中”。

推荐阶段：

- `数据归集`
- `满血研判`
- `结果校验`
- `结论整理`

这些阶段是用户可见进度语义，不要求和底层每个函数一一映射，但必须能稳定表达“系统正在推进，而不是卡死”。

#### 4.2 分析完成

只有当单AI拿到真实有效结果时才进入完成态。

完成态继续展示：

- 本局决断
- 决策机制
- 决断强度
- 展开决断详情

#### 4.3 分析失败

如果 `120` 秒窗口结束仍无有效结果，则进入失败态。

失败态必须包含：

- 标题：例如 `本轮分析未完成`
- 主说明：明确失败原因
- 辅助说明：例如 `当前还没有形成有效预测结果`
- 主按钮：`重新分析`

失败态不能继续显示：

- 建议方向
- 置信度
- 已完成研判
- 伪造的本局决断

### 5. 重新分析按钮设计

失败态卡片中新增主操作按钮：

- 文案：`重新分析`

行为：

1. 用户点击后，发起新的单AI分析请求
2. 新请求基于当前这局最新已开奖数据重新构建完整输入
3. 新一轮分析窗口重新计时 `120` 秒
4. 新一轮仍然是满血单AI正式分析，不允许降级

限制：

- 如果当前已经处于重新分析中，按钮进入禁用/加载态
- 防止同一局被用户连续快速点击触发多轮并发重试

### 6. 后端工作流设计

后端需要把单AI正式分析拆成两层概念：

#### 6.1 分析轮次

每一轮分析都有自己的：

- 开始时间
- 截止时间
- 当前阶段
- 最近失败原因
- 是否允许再次重试

#### 6.2 分析结果

只有当某一轮分析成功生成合法结果时，才写入正式 `analysis_outcome`。

如果某轮失败：

- 保留失败诊断
- 不生成成功结果
- 等待用户手动点击 `重新分析`

### 7. 记录与日志设计

记录页仍然保留所有分析尝试，不因为首页失败态而丢失信息。

至少新增两类可读日志语义：

1. `单AI满血分析在 120 秒窗口内未完成`
2. `用户手动重新发起单AI分析`

日志要求：

- 中文化
- 不直接暴露英文原始异常
- 仍保留足够的排障线索

### 8. 详情抽屉设计

如果当前没有成功结果，则详情抽屉不应继续沿用成功态结构。

规则：

- 有真实有效结果：正常展示详情抽屉
- 失败态：不展示成功决断详情
- 如需展示失败说明，应在主卡失败态中完成，不强行复用成功详情抽屉

### 9. 与旧逻辑的冲突处理

本次设计与现有单AI工作流中的几个旧约束冲突，必须明确替换：

#### 9.1 替换规则兜底接管

旧逻辑：

- 单AI失败 -> `rule_fallback` 接管

新逻辑：

- 单AI失败 -> 进入明确失败态 -> 等待用户手动重新分析

#### 9.2 替换伪完成展示

旧逻辑风险：

- 解析失败/超时后仍可能出现看似完成的展示

新逻辑：

- 没有有效单AI结果，就绝不允许进入完成态

#### 9.3 替换“尽快出结果优先”

旧逻辑倾向：

- 为了让流程继续而尽量快给结果

新逻辑：

- 准确率和满血分析优先
- 允许 `120` 秒认真完成分析
- 超时则说清原因，不造假、不降级

## 测试策略

本次至少补齐以下回归：

1. 单AI模式下分析窗口超过 `120` 秒后进入失败态，而不是规则兜底
2. 失败态不生成成功 `analysis_outcome`
3. 失败态前端不显示“已完成研判 / 本局决断 / 决断强度”
4. 失败态前端显示清晰中文失败说明
5. 点击 `重新分析` 后会启动新一轮 `120` 秒分析
6. 重试期间按钮进入加载/禁用，防止重复触发
7. 新一轮成功后，首页恢复为正常成功态
8. 记录页保留失败与手动重试日志
9. 日志与失败说明不再出现英文技术黑话

## 风险与边界

### 风险 1：用户连续重试导致上游压力放大

控制方式：

- 对单局重试按钮加加载态和并发限制
- 同一时刻仅允许一轮有效重试

### 风险 2：失败态与旧前端状态混用

控制方式：

- 明确把失败态从成功态结构中分离
- 不再复用“已完成研判”的展示壳子

### 风险 3：仍然无法做到数学上的 100% 必出结果

说明：

- 本次设计追求的是“100% 坚持满血单AI、不降级”
- 不是“100% 保证每轮一定成功”
- 当上游服务在 `120` 秒窗口内持续异常时，系统必须说真话并进入失败态

## 结论

本次单AI模式需要升级成新的强约束定义：

- 只允许开奖后启动
- 只允许满血单AI正式分析
- 单轮窗口固定 `120` 秒
- 不允许降级
- 失败后必须把原因讲清楚
- 由用户点击 `重新分析` 手动开启新一轮 `120` 秒满血分析

这套设计牺牲的是“永远给一个结果”的幻觉，换来的是：

- 更高的结果真实性
- 更高的可解释性
- 更符合用户对“准确率优先”的核心要求
