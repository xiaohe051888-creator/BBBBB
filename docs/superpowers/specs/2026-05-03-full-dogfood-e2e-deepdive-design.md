# 全量深挖：真实用户流程 Dogfood + E2E 回归体系设计

## 目标

- 把“正常使用才会遇到的问题”变成可重复的自动化回归：一次修复后不再复发
- 覆盖从 0 到 100 的关键链路：配置→启用→数据→下注→开奖→结算→日志→维护→异常恢复
- 默认跑“稳定回归”（本地 mock 上游），同时保留可选的“真实上游”（外网 AI）验证用例

## 范围与成功标准

### 关键业务链路（必须覆盖）

- 管理员登录/退出登录、token 过期后的体验
- 模式切换：规则 / 单AI / 3AI（含门禁：未配置/未测试/已测试）
- AI 配置：保存、测试连通性、错误提示、保存后 UI 状态一致性
- 上传数据：上传文件、解析、落库、异常文件处理、重复上传、清库/重置 boot
- 下注：下单、查看注单、分页/筛选、错误场景（余额不足/参数非法/重复提交）
- 开奖/结算：开奖接口、结算后状态变化、日志记录、排行榜/统计（若有）
- 日志：实时推送、刷新、导出、权限、空状态、性能（大数据量）
- 任务：后台任务列表、启动/停止任务、任务失败提示、任务日志可用性
- 运维：维护统计、清理任务、告警列表与确认、权限校验

### 回归稳定性标准（必须达成）

- E2E 可在干净环境一键跑通（不依赖人工点击）
- 用例隔离：每条用例可重跑、不依赖上条用例残留数据
- 失败时有证据：Playwright 自动保留截图/视频/trace，报告能直接定位问题

## 测试分层与策略

### 1) 浏览器端到端（Playwright）— 真实用户视角

- 默认入口：`npm run e2e`
- 覆盖 UI 流程 + 下载/导出 + WebSocket 实时行为
- 默认稳定回归（本地 mock 上游）：
  - `E2E_USE_MOCK=1`（默认）：把单AI/3AI 上游 base_url 指向本地 mock（OpenAI 兼容）
  - `E2E_REAL_AI=1`：启用“真实外网连通性”可选用例（不作为默认门禁，避免网络/额度波动导致误报）

### 2) API 冒烟（后端脚本/单测）— 速度快、定位准

- 覆盖关键接口：登录、配置、测试连通性、切模式、上传、下注、开奖、日志、维护
- 与 E2E 互补：E2E 更像用户，API 冒烟更容易定位服务端原因

### 3) “测试专用重置/造数”能力（受环境开关保护）— 让深挖可重复

新增仅测试环境可用的管理接口（或复用已有接口组合）：

- 清空/重置：
  - 清库：games/bets/logs/versions/tasks（按需可选）
  - 重置 boot number / 清理当日缓存
- 造数：
  - 生成 N 局游戏记录
  - 生成下注记录（含不同金额/状态/输赢）
  - 生成日志（用于大数据量分页/导出/性能）
- 快速切换：
  - 切换预测模式（rule / single_ai / ai）
  - 可选：把 AI 配置设置为指定 provider/model/base_url（便于 mock/真实切换）

保护策略：

- 仅在 `E2E_TESTING=1`（或类似）时启用路由
- 仍需管理员 token
- 不输出任何敏感信息

## 用例矩阵（第一阶段）

### E2E-001：管理员登录→配置单AI→测试→启用→日志刷新/导出

- 当前已具备雏形，用于打通 E2E 基础设施

### E2E-002：规则模式→上传→下注→开奖→结算→日志验证

- 使用测试重置接口生成/上传数据
- 下注/开奖/结算后，验证 UI 状态与日志一致

### E2E-003：token 过期/401→回到可用状态

- 造一个过期 token（或主动清理 token）
- 验证不会闪屏/循环跳转，且关键操作会提示需要登录

### E2E-004：3AI 模式门禁与错误提示链路

- 未配置→按钮禁用 + 明确提示
- 配置但未测试→提示先测试
- 测试通过→允许启用

### E2E-005：后台任务与维护面板

- 任务列表能加载
- 触发一次清理/维护动作，验证结果提示、列表刷新

## 输出与跟踪

- 每次“全量深挖”输出：
  - 自动化报告（Playwright html + trace）
  - 新发现问题清单（按严重程度 + 复现步骤 + 建议修复点）
- 每修复一个问题：
  - 追加一个回归用例或断言，保证不复发

## 非目标（暂不做）

- 不在此阶段做性能压测平台化（但会加入基础性能断言与大数据量场景）
- 不做生产环境对接的秘密管理改造（只保证测试过程不泄露密钥）

## 风险与应对

- 外网 AI 不稳定导致误报：默认 `E2E_USE_MOCK=1`；真实外网用例仅在 `E2E_REAL_AI=1` 时跑
- 数据量大导致 E2E 超时：通过重置/造数接口控制规模；必要时分组执行
- 端到端不易定位：E2E 失败同时抓取对应 API response 与后端日志片段（可选增强）
