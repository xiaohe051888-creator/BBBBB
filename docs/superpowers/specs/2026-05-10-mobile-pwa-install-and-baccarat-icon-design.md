# 手机可安装 PWA 与智能分析百家乐图标设计

## 1. 背景

当前前端为标准 Vite + React 单页应用，尚未具备手机浏览器标准安装能力。

现状缺口：

- 没有 `manifest.webmanifest`
- 没有 `service worker`
- 没有安装相关 meta
- 没有独立窗口打开配置
- 没有用于手机桌面图标的成套图标资源

本次目标是在不改动现有业务逻辑的前提下，把前端补成可安装的 PWA 外壳，并设计一套“智能分析百家乐”主题图标。

## 2. 目标

本次设计要达成以下结果：

- 安卓浏览器支持标准安装为桌面 App
- iPhone Safari 支持“添加到主屏幕”
- 安装后默认以独立窗口打开，而不是浏览器标签页
- 提供稳定、清晰、可识别的手机桌面图标
- 对静态资源做保守缓存，提升首屏稳定性
- 保持实时接口和 WebSocket 不被离线缓存污染

## 3. 非目标

以下内容不在本次范围内：

- 不做推送通知
- 不做离线登录
- 不做离线分析、离线下注或离线实时数据
- 不改变后端接口行为
- 不重构现有路由或页面结构

## 4. 方案比较

### 方案 A：标准 PWA

内容：

- 增加 `manifest.webmanifest`
- 增加 `service worker`
- 注册安装壳
- 提供安装图标和启动页配置

优点：

- 安卓安装体验最完整
- iPhone 兼容性最好
- 后续可平滑扩展安装提示与版本更新提示

缺点：

- 比单纯快捷方式接入多一层缓存与注册逻辑

结论：

- 采用本方案

### 方案 B：仅做快捷方式外观

内容：

- 只补 meta 和图标，不加 service worker

优点：

- 工作量最小

缺点：

- 安卓标准安装能力不完整
- 后续还需要补做标准 PWA

结论：

- 不采用

### 方案 C：重离线缓存 PWA

内容：

- 大范围缓存静态页与接口响应

优点：

- 离线可打开更多内容

缺点：

- 对实时系统风险大
- 容易展示过期状态、过期分析结果或过期登录态

结论：

- 不采用

## 5. 最终设计

采用“标准 PWA + 保守缓存”的方式接入。

核心原则：

- 安装能力完整
- 只缓存稳定静态资源
- 不缓存实时接口结果
- 不影响现有业务流程

用户可感知行为：

- 安卓浏览器可触发安装 App
- iPhone 可手动添加到主屏幕
- 从桌面打开时显示为独立应用窗口
- 图标与启动画面风格统一

## 6. 资源与文件设计

### 前端入口

需要在前端入口注册 `service worker`。

涉及文件：

- `frontend/src/main.tsx`

设计要求：

- 仅在生产环境注册
- 注册失败不影响页面正常打开
- 代码保持简单，不引入业务耦合

### 安装清单

新增标准 `manifest.webmanifest`。

涉及文件：

- `frontend/public/manifest.webmanifest`

关键字段：

- `name`: 百家乐智能分析
- `short_name`: 智能百家乐
- `display`: `standalone`
- `theme_color`: 深色主题
- `background_color`: 深色主题
- `start_url`: `/`
- `icons`: 至少提供 `192x192` 与 `512x512`

### iPhone 兼容

需要在 `index.html` 中补充安装相关 meta。

涉及文件：

- `frontend/index.html`

至少包含：

- `apple-mobile-web-app-capable`
- `apple-mobile-web-app-status-bar-style`
- `apple-mobile-web-app-title`
- `theme-color`
- `manifest` 引用
- `apple-touch-icon`

### Service Worker

新增一个最小但安全的 `service worker`。

建议文件：

- `frontend/public/sw.js`

缓存策略：

- 预缓存：应用壳静态资源、图标、favicon、manifest
- 运行时缓存：构建产物中的 js/css/image
- 不缓存：`/api/*`
- 不缓存：`/ws`
- 导航请求优先网络，失败时可回退到缓存入口页

原因：

- 该系统高度依赖实时状态
- 不能让旧分析结果、旧登录状态或旧告警被缓存误用

## 7. 图标设计

### 设计目标

图标需要同时表达三层含义：

- 百家乐
- 智能分析
- 高级、专业、可安装 App 质感

### 设计方向比较

#### 方向 A：扑克牌 + 数据线

优点：

- 赌博主题辨识度高

缺点：

- 容易显得廉价
- 小尺寸下细节容易糊

结论：

- 不采用

#### 方向 B：筹码 + 折线图

优点：

- 能体现“游戏 + 分析”

缺点：

- 容易接近通用金融 App 图标
- 百家乐识别度一般

结论：

- 作为备选

#### 方向 C：百家乐对立意象 + 智能分析环

设计构成：

- 深色圆角底
- 中心使用 `B` / `P` 或庄闲对立语义的核心符号
- 外圈加入雷达环、扫描弧或分析轨迹
- 局部点缀金色高光，体现专业和结果判断

优点：

- 主题集中
- 小尺寸下识别度高
- 更像高端工具类 App，而不是娱乐场海报

结论：

- 采用本方向

### 最终图标规范

图标主风格：

- 深蓝黑底色
- 蓝紫分析光效
- 金色结果焦点
- 几何化、简洁化、桌面图标优先

图标视觉元素：

- 主体：抽象化百家乐对立符号
- 辅助：分析弧线、雷达圈、上升信号
- 避免：人物、复杂牌面、大量文字、廉价赌场光效

图标输出要求：

- 生成主图标 SVG
- 导出至少 `192x192`、`512x512`
- 兼容 `maskable` 场景，尽量保证主体居中

## 8. 技术实现边界

为了减少风险，本次采用手工可控接入，而不是一次性引入复杂离线框架。

实现方式：

- 可以直接使用原生 `service worker`
- 也可以使用 `vite-plugin-pwa` 生成标准文件

最终建议：

- 使用 `vite-plugin-pwa`

原因：

- 与 Vite 集成自然
- 生成 manifest、注册逻辑和构建产物更稳定
- 比手写缓存清单更不容易漏资源

同时保留以下约束：

- 明确排除 `/api` 和 `/ws`
- 缓存策略保持保守
- 不引入推送或后台同步功能

## 9. 测试与验收

### 功能验收

需要验证以下结果：

- 构建后页面包含 `manifest`
- `service worker` 成功注册
- 安卓浏览器可出现安装入口
- iPhone Safari 可“添加到主屏幕”
- 从桌面打开后表现为独立窗口
- 升级后刷新不会卡死在旧壳

### 风险验收

需要确认：

- `/api` 请求不会被旧缓存返回
- WebSocket 不被 service worker 影响
- 登录后刷新状态不出现异常回滚
- 前端路由在独立窗口下仍然正常

### 回归范围

- 首页
- 登录页
- 日志页
- 移动端主要导航

## 10. 实施顺序

建议按以下顺序实施：

1. 引入 PWA 插件并补构建配置
2. 新增 manifest 与安装 meta
3. 注册 service worker
4. 设计并生成图标资源
5. 构建并检查产物
6. 在手机浏览器中实际安装验证

## 11. 风险与应对

风险一：缓存导致旧页面残留

应对：

- 导航优先网络
- API 不缓存
- 使用构建哈希资源

风险二：图标小尺寸不清晰

应对：

- 图标几何化
- 减少文字和复杂牌面细节
- 先以桌面尺寸识别为优先标准

风险三：iPhone 行为与安卓不完全一致

应对：

- 把 iPhone 目标定义为“可添加到主屏幕”
- 不承诺与安卓完全一致的安装提示

## 12. 结论

本次采用“标准 PWA + 保守缓存 + 智能分析百家乐主题图标”的方案。

该方案能在不影响实时业务可靠性的前提下，给手机浏览器提供接近原生 App 的安装体验，并用统一的深色高质感图标强化产品识别度。
