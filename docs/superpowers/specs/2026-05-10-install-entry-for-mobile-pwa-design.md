# 首页安装 App 入口设计

## 1. 背景

当前站点已经具备基础 PWA 安装能力：

- 已有 `manifest.webmanifest`
- 已有 `service worker`
- 已有 Android 安装所需基础条件
- 已有 iPhone Safari “添加到主屏幕”所需 meta

但用户实际打开前端链接后，没有看到明显的安装提示，导致“已支持安装”和“用户感知可安装”之间存在落差。

浏览器的原生安装提示并不稳定：

- Android Chrome 不一定主动弹出安装浮层
- 一些浏览器只在菜单或地址栏提供安装入口
- iPhone Safari 本身不会像 Android 那样给出标准安装弹窗

因此需要在前端补充一个轻量、明确、不中断使用流程的“安装 App”入口。

## 2. 目标

本次设计要达成以下结果：

- 用户进入首页后，能明显感知站点支持安装为 App
- Android 浏览器在满足条件时可从前端按钮直接触发安装
- iPhone Safari 能得到明确的“添加到主屏幕”操作指引
- 已安装状态下不再显示该入口
- 提示尽量轻，不破坏首页已有顶部布局

## 3. 非目标

以下内容不在本次范围内：

- 不做首次强打断式全屏弹窗
- 不做推送权限引导
- 不做复杂安装教程页面
- 不修改现有核心业务逻辑
- 不增加新的路由页面

## 4. 方案比较

### 方案 A：首页轻量安装入口

内容：

- 首页显示单行轻量安装入口
- Android 直接调用浏览器安装流程
- iPhone 显示“分享 -> 添加到主屏幕”引导

优点：

- 用户能立刻知道“这里可以安装”
- 不依赖浏览器自己是否弹提示
- 和现有首页结构兼容性最好

缺点：

- 需要补一个前端状态判断层

结论：

- 采用本方案

### 方案 B：静态文案提示

内容：

- 首页仅显示一段说明文字，不提供交互按钮

优点：

- 成本最低

缺点：

- 用户仍然不知道下一步怎么装
- 转化率低

结论：

- 不采用

### 方案 C：首次访问强弹窗

内容：

- 首次进入首页就弹出安装引导弹层

优点：

- 安装提示最明显

缺点：

- 打断主流程
- 首页顶部目前已做过“减干扰”优化，不适合再加强提醒

结论：

- 不采用

## 5. 最终设计

采用“首页轻量安装入口”方案。

整体原则：

- 能被看见
- 不抢主内容
- 已安装自动消失
- Android 与 iPhone 分流处理

用户感知流程：

- Android 可安装时：首页出现“安装 App”按钮
- 用户点击后：直接触发浏览器安装流程
- iPhone Safari：首页出现“安装到桌面”按钮
- 用户点击后：出现一个轻量说明层，告诉用户“点击分享 -> 添加到主屏幕”

## 6. 入口位置

推荐挂载位置：

- 首页 `DashboardPage` 顶部状态区域附近
- 放在已有顶部信息流和轻提示组件附近
- 采用单行、低视觉噪声样式，不做大面积横幅

原因：

- 首页是主要访问入口
- 安装意图通常在首次使用时出现
- 放在顶部更容易被看到，但需要控制视觉权重

## 7. 显示逻辑

安装入口只在满足以下条件时显示：

- 当前不是独立安装模式
- 当前不是已添加到主屏幕的 iPhone Web App 模式
- 当前浏览器支持安装或属于需要引导的 iPhone Safari
- 用户没有在短期内主动关闭过安装提示

判定维度：

- `window.matchMedia('(display-mode: standalone)')`
- `navigator.standalone`
- `beforeinstallprompt`
- `userAgent` + Safari / iPhone 条件识别
- 本地存储的关闭时间戳

隐藏条件：

- 已安装
- 处于 standalone 模式
- 用户手动关闭入口后，在设定周期内不再展示

建议的本地关闭冷却时间：

- 7 天

## 8. Android 行为

Android 侧采用标准浏览器安装流程。

设计方式：

- 监听 `beforeinstallprompt`
- 阻止浏览器默认自动行为
- 保存事件对象
- 当用户点击“安装 App”按钮时：
  - 调用 `prompt()`
  - 读取 `userChoice`
  - 根据结果关闭或保留入口

结果处理：

- 用户接受安装：入口立即隐藏
- 用户取消安装：入口可以保留，但不重复强提醒

## 9. iPhone 行为

iPhone Safari 不支持标准 `beforeinstallprompt`。

因此采用引导说明方式：

- 按钮文案为“安装到桌面”
- 点击后打开轻量说明层

说明层内容只保留必要信息：

1. 点击 Safari 下方或上方的分享按钮
2. 选择“添加到主屏幕”
3. 完成后即可像 App 一样从桌面打开

说明层样式要求：

- 轻量，不做重型模态
- 可关闭
- 文案简短，控制在 3 步以内

## 10. 交互与视觉

### 入口形态

采用轻量胶囊按钮或小卡片按钮。

建议文案：

- Android：`安装 App`
- iPhone：`安装到桌面`

附属说明可选：

- `像 App 一样打开`

### 视觉要求

- 颜色与当前深色主题一致
- 保持弱高亮，不用强警告色
- 尺寸不要压缩主要业务信息
- 移动端优先保证单行可读

### 关闭能力

入口旁提供一个轻量关闭操作，或在说明层中提供关闭能力。

关闭后：

- 写入本地存储
- 7 天内不再显示

## 11. 技术拆分

建议拆成以下几个独立单元：

### `useInstallPromptState`

职责：

- 统一判断当前是否应显示入口
- 识别 Android / iPhone / 已安装状态
- 管理 `beforeinstallprompt`
- 管理本地关闭状态

### `InstallAppEntry`

职责：

- 渲染首页安装入口按钮
- 根据平台展示不同文案
- 处理按钮点击逻辑

### `InstallAppGuide`

职责：

- 为 iPhone 展示轻量说明层
- 显示 3 步安装说明
- 支持关闭

这种拆分能保证判断逻辑、入口 UI、说明层 UI 各自独立，后续更容易测试和维护。

## 12. 测试要求

需要覆盖以下行为：

- Android 且存在 `beforeinstallprompt` 时显示 `安装 App`
- 点击 Android 按钮时触发 `prompt()`
- iPhone Safari 显示 `安装到桌面`
- 点击 iPhone 按钮后出现安装说明层
- standalone 模式下入口不显示
- 用户关闭后，短期内入口不再出现

测试类型：

- 组件测试
- Hook 行为测试
- 必要的页面挂载测试

## 13. 风险与应对

风险一：不同浏览器对安装能力支持不一致

应对：

- 入口逻辑以能力检测为主
- iPhone 走引导，不强求标准安装事件

风险二：入口干扰首页顶部布局

应对：

- 采用轻量单行按钮
- 不做重横幅
- 尽量与现有顶部结构对齐

风险三：用户反复看到入口产生打扰

应对：

- 提供关闭能力
- 本地冷却 7 天
- 安装后自动隐藏

## 14. 结论

本次采用“首页轻量安装入口”方案。

它能解决“已经支持安装但用户没看到提示”的问题，同时保持首页视觉克制，兼顾 Android 与 iPhone 的平台差异，是当前阶段最合适的补齐方式。
