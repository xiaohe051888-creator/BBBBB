# 单AI正式预测提示词重构设计

## 背景

当前单AI模式已经固定为 DeepSeek `V4 Pro + thinking=enabled`，并且解析器、状态机、规则兜底、前端展示都已经按“结果真实性”收口。

现在剩下的关键问题不是“失败时怎么说真话”，而是“如何减少模型输出漂移，让单AI更稳定地完成它的真实任务”。

这个真实任务必须被明确写死：

- 单AI正式分析提示词的目的，是基于当前靴历史、五路走势、五路特征和错题上下文，推理并预测**下一局**是 `庄` 还是 `闲`
- 它不是为了输出开放式讲解
- 它不是为了写长篇分析
- 它不是为了描述不确定性本身
- 它必须服务于“下一局庄闲预测”

## 目标

本次重构的目标有四个：

1. 强化单AI提示词的唯一目标：预测下一局 `庄 / 闲`
2. 提高 DeepSeek 正式输出的结构稳定性，减少字段漂移和非 JSON 输出
3. 保留推理能力，但把推理约束在结构化字段中，避免自由扩写污染返回
4. 在信号冲突时仍然必须选边，只能通过 `confidence` 表达不确定性，不能放弃预测

## 非目标

本次不处理以下事项：

- 不改动单AI运行时配置（仍然是 DeepSeek `V4 Pro + thinking=enabled`）
- 不改动解析器有效结果契约
- 不改动规则兜底的接管条件
- 不引入“无法判断”或“观望”这类第三种预测结果

## 方案对比

### 方案 A：维持现状，只补少量措辞

做法：

- 在现有默认提示词上继续加几句“只输出 JSON”
- 继续保留当前混合式长提示结构

优点：

- 改动最小

缺点：

- 目标不够聚焦
- 输出契约与分析要求仍然混在一起
- 对 `thinking` 模式模型的结构稳定性帮助有限

结论：

不采用。

### 方案 B：极简格式提示词

做法：

- 只保留“预测下一局庄闲 + 输出固定 JSON”
- 尽量删除分析说明

优点：

- 结构最稳定

缺点：

- 容易削弱五路推理质量
- 模型可能只给机械结论，推理字段信息密度下降

结论：

不单独采用。

### 方案 C：契约优先的预测型提示词

做法：

- 把提示词拆成清晰层次：角色目标、决策规则、输出契约、输入数据
- 明确写死单AI的核心任务是预测下一局 `庄 / 闲`
- 明确冲突盘面下也必须选边
- 允许内部深度思考，但最终只能输出严格 JSON

优点：

- 同时兼顾预测目标、结构稳定性和推理质量
- 更适合当前已经严格化的解析器与状态机

结论：

采用方案 C。

## 设计

### 1. 角色目标重写

默认提示词开头不再使用宽泛的“分析引擎”描述，而是改成更窄的目标声明：

- 你正在预测**下一局**百家乐结果
- 你只能在 `庄` 和 `闲` 中二选一
- 你的任务不是讨论是否预测，而是完成预测

这样做的目的，是避免模型把任务理解成“开放式走势评论”，而不是“需要产出可下注方向的预测决策”。

### 2. 决策规则显式化

提示词需要把决策顺序写清楚，而不是把输入信息全部平铺后让模型自由发挥。

推荐顺序：

1. 先阅读五路特征摘要
2. 再核对五路原始点位
3. 再结合历史结果与错题上下文
4. 最终只回答“下一局更偏庄还是更偏闲”

同时增加明确约束：

- 即使信号冲突，也必须选 `庄` 或 `闲`
- 不允许输出“无法判断”“继续观察”“等待更多数据”
- 如果把握不高，只能降低 `confidence`

### 3. 输出契约独立成段

当前提示词把输出要求混在正文里，本次改成单独的“输出契约”区块。

契约内容必须明确：

- 最终输出只能是一个 JSON 对象
- 不能输出 Markdown
- 不能输出代码块
- 不能输出解释性前缀或结尾
- 必须包含：
  - `final_prediction`
  - `confidence`
  - `bet_tier`
  - `summary`
  - `reasoning_points`
  - `reasoning_detail`

并且字段语义进一步固定：

- `final_prediction`: 只能是 `庄` 或 `闲`
- `confidence`: `0` 到 `1`
- `summary`: 一句话说明为什么偏向这个方向
- `reasoning_points`: 只保留高价值要点，避免冗长
- `reasoning_detail`: 解释“为什么预测下一局是这个方向”

### 4. 推理文本压缩

本次不削弱模型推理，只压缩外显文本。

具体做法：

- 保留“可以先内部深度思考”
- 但要求外显推理只能落在既定字段中
- `reasoning_points` 限制为少量高价值要点
- `reasoning_detail` 要围绕“下一局庄闲预测依据”展开，不能泛化成长篇走势评论

这样做的目的，是减少结构污染，同时保留足够的解释性，方便后端和前端消费。

### 5. 输入数据分区

默认提示词中的输入数据需要按职责分区，避免模型把整段内容当成同权重噪音。

建议区块：

- 局面元信息：
  - `boot_number`
  - `game_number`
  - `consecutive_errors`
- 历史结果：
  - `game_history`
- 高层特征：
  - `road_features`
- 原始五路：
  - `road_data`
- 纠偏上下文：
  - `mistake_context`

这种组织方式的目的，是让模型先看“摘要层”，再看“证据层”，最后看“纠偏层”。

### 6. 模板化能力保留

当前系统允许学习版本把 `prompt_template` 注入正式分析链路，这个能力保留。

但默认提示词的职责要改变：

- 默认提示词成为稳定、严格、可解析的基线模板
- 自定义或学习模板必须继续兼容当前字段契约

换句话说，默认模板不再追求“花样分析风格”，而是作为稳定生产基线存在。

### 7. 测试策略

本次提示词重构虽然主要改文案，但它本质上影响运行行为，因此要补回归。

至少覆盖：

1. 默认提示词仍明确要求预测下一局 `庄 / 闲`
2. 默认提示词明确要求即使冲突也必须选边
3. 默认提示词明确要求只输出严格 JSON
4. 现有解析契约测试继续保持通过

如果当前测试结构适合，也可以增加对 `_build_prompt()` 文本内容的断言，锁住这些关键约束。

## 预期结果

重构完成后，单AI正式提示词应具备以下特征：

- 任务目标更单一：只服务于下一局庄闲预测
- 输出更稳定：减少非 JSON、少字段、漂移字段
- 解释更聚焦：围绕预测结论，不再泛化扩写
- 冲突盘面更符合业务：仍然必须给出下注方向

## 风险与控制

### 风险 1：提示词过严，推理质量下降

控制方式：

- 保留内部深度思考许可
- 只压缩外显输出，不压缩内部推理空间

### 风险 2：学习生成模板与默认契约偏离

控制方式：

- 默认模板继续作为生产基线
- 解析器仍然是最终验收门

### 风险 3：模型在极端盘面下仍试图输出观望

控制方式：

- 在提示词里显式禁止“无法判断/观望”
- 用测试锁住“必须选边”的核心约束

## 结论

本次单AI正式提示词需要重构，而且必须围绕一个核心原则展开：

**单AI的任务不是泛泛分析走势，而是基于当前证据推理并预测下一局到底是庄还是闲。**

因此本次采用“契约优先的预测型提示词”方案，把目标、决策、输出和输入分层，作为后续实现的唯一设计基线。
