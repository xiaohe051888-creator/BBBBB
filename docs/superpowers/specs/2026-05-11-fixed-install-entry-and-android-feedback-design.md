# 固定安装入口与安卓安装反馈设计

## 1. 背景

当前首页已经接入安装入口，但实际使用中仍有两个核心问题：

- 安装入口会因为浏览器能力判断或本地关闭冷却而隐藏
- 安卓 Chrome 用户点击 `安装 App` 后，在 `beforeinstallprompt` 不可用时没有任何反馈

这两个问题会直接造成用户感知落差：

- 用户希望首页始终有明确的安装入口
- 用户点击按钮后，希望无论能不能直接安装，都能得到明确反馈，而不是“没反应”

本次设计是在已有 PWA 安装壳和首页安装入口基础上做一次交互修正，目标是让入口稳定可见、点击必有结果、已安装状态仍然自动隐藏。

## 2. 目标

本次设计要达成以下结果：

- 只要当前不是 App 独立模式打开，首页安装按钮固定存在
- 已安装或已用 App 版本打开时，安装入口不显示
- 安卓 Chrome 有原生安装能力时，点击按钮直接拉起安装流程
- 安卓 Chrome 当前拿不到安装能力时，点击按钮立即给出中文提示
- iPhone Safari 继续提供“添加到主屏幕”引导
- 去掉会导致入口消失的关闭冷却逻辑

## 3. 非目标

以下内容不在本次范围内：

- 不修改 PWA `manifest`、`service worker` 或图标资源
- 不新增独立安装说明页面
- 不兼容所有内嵌浏览器的真实安装能力差异
- 不保证所有安卓浏览器都能弹出系统安装框
- 不增加推送、离线权限或通知引导

## 4. 问题归因

当前“点击安装 App 没反应”的根因不是按钮失效，而是前端缺少“不可直接安装时的明确反馈”。

现有逻辑大致如下：

- `beforeinstallprompt` 事件存在时，安卓按钮可以触发 `prompt()`
- 事件不存在时，`triggerInstall()` 返回 `unavailable`
- 调用方没有针对 `unavailable` 给出中文提示

同时，现有显示逻辑还叠加了两个隐藏条件：

- 非 `android` / `ios` 平台时入口直接隐藏
- 用户点过“稍后”后，入口在 7 天内不再出现

这与“按钮固定存在”的新要求冲突，因此本次必须一起调整。

## 5. 方案比较

### 方案 A：固定显示入口 + 分平台反馈

内容：

- 只要未安装，就固定显示安装入口
- 安卓可安装时直接安装
- 安卓不可安装时弹出中文说明
- iPhone 继续走引导说明

优点：

- 最符合“按钮固定存在”的要求
- 用户点击后一定有结果，不会再出现“没反应”
- 改动范围集中在首页入口和状态判断层

缺点：

- 需要扩展当前平台状态和反馈组件

结论：

- 采用本方案

### 方案 B：固定显示入口 + 不可安装时禁用按钮

内容：

- 按钮始终显示
- 当前不可安装时按钮置灰

优点：

- 实现简单

缺点：

- 用户更容易误判为系统坏了
- 无法解释为什么当前不能安装

结论：

- 不采用

### 方案 C：固定显示入口 + 永远只给说明

内容：

- 无论是否支持原生安装，都只弹说明层

优点：

- 交互统一

缺点：

- 丢失安卓一键安装体验
- 用户操作路径变长

结论：

- 不采用

## 6. 最终设计

采用“固定显示入口 + 分平台反馈”方案。

核心原则：

- 入口常驻，但不打扰
- 点击必有反馈
- 能直接安装时不增加额外步骤
- 不能直接安装时给出中文说明
- 已安装状态仍然自动隐藏

用户感知流程：

- 已安装 App 打开：不显示安装入口
- 安卓 Chrome 且支持安装：显示 `安装 App`，点击后直接拉起系统安装框
- 安卓 Chrome 但当前不支持直接安装：显示 `安装 App`，点击后弹出中文说明层
- iPhone Safari：显示 `安装到桌面`，点击后打开三步引导层

## 7. 显示逻辑

### 显示条件

安装入口显示条件调整为：

- 当前不是 `standalone` 独立窗口模式
- 当前不是 iPhone 已添加到主屏幕后的 Web App 模式

只要满足以上两条，就显示安装入口。

### 隐藏条件

安装入口仅在以下条件下隐藏：

- `window.matchMedia('(display-mode: standalone)')` 为真
- `navigator.standalone` 为真

### 移除的旧逻辑

以下旧逻辑与本次目标冲突，需要移除：

- `7 天关闭冷却`
- “用户点稍后后整个入口消失”
- “平台识别为 `none` 时直接隐藏入口”

## 8. 平台状态设计

当前平台状态不能再只使用 `android | ios | none`，否则无法表达“入口固定显示但当前只能给说明”的场景。

建议扩展为更贴近交互的状态：

- `android-ready`
- `android-help`
- `ios`

语义如下：

- `android-ready`：安卓浏览器已拿到 `beforeinstallprompt`
- `android-help`：安卓浏览器环境成立，但当前没有可直接调用的安装事件
- `ios`：iPhone Safari，需要走“添加到主屏幕”说明

若未来出现桌面浏览器或非目标环境，也不再返回“完全隐藏”，而是由调用层决定是否落到 `android-help` 风格说明或保留现有默认行为。

本次优先保证用户已确认的主路径：

- 安卓 Chrome
- iPhone Safari

## 9. 安卓行为

### 安卓可直接安装

当已捕获 `beforeinstallprompt` 时：

- 按钮文案显示为 `安装 App`
- 点击按钮后调用 `prompt()`
- 读取 `userChoice`
- 若用户接受安装，入口立即隐藏
- 若用户取消安装，入口继续保留

### 安卓不可直接安装

当处于安卓环境但没有可用 `beforeinstallprompt` 时：

- 按钮文案仍显示为 `安装 App`
- 点击后打开一个中文说明层，而不是无反应

说明层需要明确传达：

1. 当前浏览器暂时没有返回安装能力
2. 请继续在浏览器中正常打开当前站点，不要用内嵌浏览器
3. 可以尝试使用 Chrome 菜单中的“安装应用”或“添加到主屏幕”
4. 如果仍无入口，说明当前浏览器会话暂不支持直接拉起安装

建议文案风格：

- 简短
- 全中文
- 以“为什么现在没弹”和“你下一步可以怎么做”为主

## 10. iPhone 行为

iPhone Safari 保持现有交互方向，但入口改为固定显示。

行为要求：

- 按钮文案为 `安装到桌面`
- 点击后打开轻量说明层
- 说明层保留 3 步：
  1. 点击 Safari 分享按钮
  2. 选择“添加到主屏幕”
  3. 完成后从桌面像 App 一样打开

不再提供“关闭后 7 天隐藏”的逻辑。

## 11. UI 设计

### 入口形态

保留当前单行轻量入口样式：

- 主标题：`安装 App` 或 `安装到桌面`
- 副文案：`像 App 一样打开`

### 操作按钮

建议保留单一主操作按钮，不再保留“稍后”隐藏逻辑。

原因：

- 用户已经明确要求入口固定存在
- “稍后”与“固定存在”互相冲突
- 去掉隐藏按钮后，用户心智更稳定

### 说明层

新增或扩展一个安卓说明层，和 iPhone 引导层保持同级轻量风格：

- 不使用重型全屏弹窗
- 允许关闭
- 文案清晰
- 安卓与 iPhone 说明分开，避免混淆

## 12. 技术拆分

建议沿用现有拆分，并做以下职责调整。

### `useInstallPromptState`

职责调整为：

- 判断当前是否已安装
- 返回新的平台状态
- 管理 `beforeinstallprompt`
- 暴露 `triggerInstall()`
- 暴露“安卓帮助层”和“iPhone 引导层”的开关

需要移除：

- 本地关闭时间戳读取
- 冷却隐藏逻辑

### `InstallAppEntry`

职责调整为：

- 始终在未安装时渲染入口
- 根据平台状态显示不同主按钮文案
- 根据平台状态决定点击行为：
  - 直接安装
  - 打开安卓说明层
  - 打开 iPhone 引导层

### `InstallAppGuide`

职责保留：

- 负责 iPhone 三步引导

### 新增 `InstallAppHelp`

建议新增安卓帮助说明层组件。

职责：

- 展示“为什么没弹安装框”
- 展示用户下一步可执行的操作
- 支持关闭

若实现上更适合复用现有说明层结构，也可以用一个组件按平台渲染不同内容，但最终交互语义必须保持清晰。

## 13. 错误处理

本次要明确覆盖两类失败反馈：

### 类型一：安装能力不可用

表现：

- `beforeinstallprompt` 不存在
- `triggerInstall()` 返回 `unavailable`

要求：

- 不能静默失败
- 必须立刻给中文说明

### 类型二：调用安装流程后用户取消

表现：

- `userChoice.outcome === 'dismissed'`

要求：

- 不视为错误
- 入口继续保留
- 不做红色报错

## 14. 测试要求

需要新增或更新以下测试：

- 未安装时入口固定显示
- standalone 模式下入口隐藏
- 安卓存在 `beforeinstallprompt` 时点击会触发 `prompt()`
- 安卓不存在 `beforeinstallprompt` 时点击会显示中文帮助说明
- iPhone 点击后显示“添加到主屏幕”引导
- 不再读取或依赖 `pwa-install-entry-dismissed-at`
- 页面挂载后，入口位于首页顶部辅助信息区域

测试层级：

- Hook 测试
- 组件测试
- 页面挂载测试

## 15. 风险与应对

风险一：固定显示后会不会干扰顶部布局

应对：

- 保持轻量单行样式
- 不改成强提示横幅
- 不增加额外次级按钮

风险二：安卓不同浏览器表现不一致

应对：

- 以安卓 Chrome 为主路径
- 直接安装失败时统一给中文说明
- 不承诺所有浏览器都能拉起系统安装框

风险三：旧冷却逻辑移除后会不会显得烦

应对：

- 通过轻量视觉降低打扰
- 已安装后自动消失
- 不再用“稍后”隐藏，但保留说明层关闭能力

## 16. 结论

本次采用“入口固定显示 + 安卓无能力时即时中文反馈 + 已安装自动隐藏”的方案。

这能同时解决两个实际问题：

- 用户找不到安装入口
- 用户点击按钮后看起来没有反应

该方案改动集中、与现有 PWA 壳兼容、实现风险低，适合作为当前安装体验的修正版设计。
