# 2026-05-09 Mobile UX Refactor Design

## 背景

当前前端已经完成过多轮移动端去壳和容器统一，但仍有 4 类明显的“小白用户体验”问题没有解决：

1. `下注记录` 页面里的 `盈亏分布` 仍沿用桌面条形布局，在手机端出现大面积空白，视觉上不像独立卡片。
2. `日志详情` 虽然已经移除了重复文案和技术字段默认展示，但默认详情仍然是纯黑白分段，阅读层级不明显。
3. `下注详情` 仍使用 `Descriptions` 单列明细表，标签和内容占位重，很多字段在手机端显得拥挤、机械。
4. `五路走势图` 的 `珠盘路` 仍采用固定宽度，导致部分屏幕上右侧空白明显，和其他路的左右铺满感不一致。

这轮目标不是继续做零散补丁，而是把这 4 个高频触点统一改成“移动端优先、信息减负、视觉层级清楚”的交互。

## 目标

### 业务目标

- 让普通用户在手机端更快读懂统计、日志和下注详情。
- 降低页面看起来“像后台表格”的感觉，提升产品感和信任感。
- 让路图区域更接近真实看盘习惯，减少无意义留白。

### 体验目标

- 默认内容先给结论，再给补充信息。
- 每个区块都应有明确卡片边界，而不是散落的文字和表格。
- 手机端优先保证单行短标签、紧凑留白、顺手扫读。

## 非目标

- 不重做整套 Dashboard 结构。
- 不修改日志含义规则和后端数据接口。
- 不引入新的交互流程或新增业务按钮。
- 不重写路图算法，只调整布局、自适应策略和展示壳层。

## 方案比较

### 方案 A：小白优先的移动端重构（采用）

- 做法：保持现有数据流和接口不变，重构四个展示层组件与其样式结构。
- 优点：风险可控，能覆盖当前用户最不满意的 4 个点，且能和已有移动端体系兼容。
- 缺点：不会一次性解决所有桌面端风格历史问题。

### 方案 B：局部 CSS 修补

- 做法：尽量不改 JSX，只通过 CSS 压缩间距、改颜色、调宽度。
- 优点：改动少。
- 缺点：治标不治本，`下注详情` 和 `日志详情` 的信息结构问题不会真正消失。

### 方案 C：整套详情页与图表系统重做

- 做法：把日志、下注、走势相关展示组件整体抽象重写。
- 优点：长期最干净。
- 缺点：范围过大，当前会拖慢交付，也会提高回归风险。

## 采用方案

采用 `方案 A`。这轮聚焦四个用户直观看到、且已经连续反馈的问题，先把默认体验做对，再考虑更大范围的统一抽象。

## 设计明细

### 1. 盈亏分布

当前位置：`frontend/src/pages/BetRecordsPage.tsx`

改造目标：

- 从“一个进度条 + 右侧标签”改成真正独立的紧凑统计卡。
- 上层先展示 `胜 / 负 / 待` 三个统计胶囊。
- 中层使用更低高度的横向比例条，减少垂直空间浪费。
- 下层补充 `胜率 / 总盈亏 / 连胜连败` 三个关键信息，形成“先概览、再判断”的阅读顺序。

具体做法：

- 给 `盈亏分布` 单独增加结构类名，避免继续依赖通用 `mobile-status-card` 的默认壳。
- `Progress` 仅承担比例表达，不再承担主信息承载。
- 将原先分散在多张统计卡中的关键结论做适度聚合，让用户不必来回看。
- 在手机端强制该卡为紧凑纵向结构，在桌面端保持可扩展。

### 2. 日志详情

当前位置：`frontend/src/pages/LogsPage.tsx`

改造目标：

- 默认详情更像“提示卡”，而不是一组无层级文本。
- 用颜色和浅背景区分 `发生了什么`、`有什么影响`、`建议怎么做`。
- 让用户一眼先看到事件性质，再看处理建议。

具体做法：

- 在详情弹窗中加入顶部摘要头区，展示优先级徽章和标题。
- 三段核心说明改成独立信息块：
  - `发生了什么`：中性信息块
  - `影响`：偏告警色信息块
  - `建议`：偏操作引导色信息块
- 不恢复技术字段默认展示，仍只放在“原始记录（高级信息）”折叠区中。
- 复制摘要按钮保留，但默认视觉焦点从“复制”转移到“看懂”。

### 3. 下注详情

当前位置：`frontend/src/pages/BetRecordsPage.tsx`

改造目标：

- 不再使用 `Descriptions` 的表单式明细。
- 改成更适合手机阅读的“下注摘要卡 + 关键信息列表”。
- 避免大量标签和内容左右撑开后各自占两排。

具体做法：

- 弹窗主体改为两段：
  - 顶部摘要：局号、状态、盈亏、下注方向四个核心信息
  - 下部列表：时间、金额、档位、开奖结果、余额变化、自适应说明
- 对 `余额变化` 这种长文本改为独立整行块，避免与短字段混排。
- 颜色继续沿用当前庄/闲、盈亏、状态体系，减少认知切换。
- 手机端下默认全部单列展示，桌面端可保持适度宽度但不回到重表格。

### 4. 五路走势图

当前位置：`frontend/src/components/roads/FiveRoadChart.tsx`、`frontend/src/components/roads/BeadRoadCanvas.tsx`

改造目标：

- 让 `珠盘路` 不再固定死宽。
- 根据容器宽度动态计算格子尺寸，在保证 6 行和 12 列可读的前提下尽量铺满宽度。
- 保持手机端 `珠盘路` 与 `大眼仔路` 分两排的现状，但减少单卡内部右侧空白。

具体做法：

- 在 `FiveRoadChart` 里去掉 `珠盘路固定像素宽度` 的布局策略，改为容器驱动宽度。
- 在 `BeadRoadCanvas` 中加入容器监听，按当前可用宽度重新计算 `cellSize`。
- 为珠盘路限定一个安全尺寸区间，避免超小屏幕上格子太大或太小：
  - 最小值保证字和圆点可辨识
  - 最大值避免在宽屏手机上显得夸张
- `大路 / 大眼仔路 / 小路 / 螳螂路` 继续使用已有自适应列数逻辑，减少不必要的算法回归。

### 样式

  - 新增 `bet-summary-card` 相关样式
  - 新增 `log-detail-*` 相关样式
  - 新增 `bet-detail-*` 相关样式
  - 新增 `bead-road-responsive` 相关样式

## 数据流

- 保持现有 React Query 和接口调用不变。
- 所有改造仅发生在展示层的结构与样式。
- 不新增后端字段，不调整已有响应格式。

## 异常与兼容策略

- 当日志数据缺失时，详情块继续显示 `-`，避免空白块导致跳动。
- 当下注记录缺少 `profit_loss` 或 `game_result` 时，摘要区使用占位文案，而不是隐藏字段。
- 当珠盘路容器宽度获取失败时，退回当前基础尺寸配置，避免渲染中断。
- 当窗口 resize 时，珠盘路重新计算尺寸并重绘，但不改变数据范围逻辑。

## 测试策略

### 前端测试

- 扩展 `frontend/src/pages/AdminMobileLayoutRegression.test.ts`
  - 锁定新的盈亏分布结构类名
  - 锁定下注详情新结构
  - 锁定五路图新的珠盘路自适应类名
- 扩展 `frontend/src/pages/LogsPage.test.tsx`
  - 锁定详情弹窗三段信息块结构
  - 锁定默认区仍不展示技术字段

### 验证方式

- 运行 `npm test` 中的目标测试文件。
- 运行 `npm run build` 确认编译通过。
- 本地启动后手动验证手机宽度下：
  - `下注记录` 页统计卡与详情弹窗
  - `日志` 页详情弹窗
  - `首页` 五路走势图

## 风险

- `BeadRoadCanvas` 改为自适应尺寸后，若尺寸范围控制不当，可能影响文字可读性。
- `LogsPage` 和 `BetRecordsPage` 弹窗结构变化较大，若样式控制不严，移动端可能再次出现过宽或过密。
- `global.css` 已承担较多移动端收口逻辑，这轮需要避免新增规则覆盖到其他卡片页。

## 实施顺序

1. 先补失败测试，锁定这轮 4 个目标点。
2. 重构 `BetRecordsPage` 的 `盈亏分布` 与 `下注详情`。
3. 重构 `LogsPage` 的 `日志详情`。
4. 重构 `FiveRoadChart` 与 `BeadRoadCanvas` 的珠盘路自适应。
5. 跑测试、构建和诊断，再做本地/线上验证。
