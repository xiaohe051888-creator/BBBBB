# 上传数据入口统一化（上传数据页 + 结束本靴/重置本靴）设计稿

日期：2026-04-28  
范围：前端（Dashboard 顶部入口 + 上传数据页面 + 确认上传弹窗）、后端（上传与换靴/重置语义对齐）  
目标：为“高频测试/回放路单/纠错覆盖”提供高效且低误触的统一上传入口

## 背景与问题

当前系统首页顶部存在“结束本靴”入口（确认弹窗 + 调用 `/games/end-boot`），但前端已不存在“上传数据页面”。同时，测试场景需要在不换靴的情况下用一批新数据覆盖当前靴并重新开始验证。

## 目标

1. 首页顶部只保留一个入口按钮：上传数据。
2. 上传数据在独立页面完成，复用“旧上传页”的录入效率：
   - 键盘快速输入：1=庄、2=闲、3=和
   - 珠盘路格子：固定 6×12=72 格，点击循环 空→庄→闲→和→空
   - 双向同步：键盘输入即时填充到格子；格子点击即时回写到局序列
3. 确认上传时可选择动作：
   - 重置本靴（覆盖本靴）：boot_number 不变，清空本靴全部数据与本靴微学习后写入新数据，从第 1 局重新开始
   - 结束本靴（开启新靴）：boot_number +1（新靴），保留旧靴历史后写入新数据，从第 1 局开始
4. 当选择“结束本靴”时，额外提供是否执行深度学习（end_boot）的开关：
   - 默认开启（严谨成长链路）
   - 可关闭（快速实验模式）
5. 余额处理在上传确认时可选：
   - 保留当前余额（默认）
   - 重置余额到 20000
6. 防误操作：不要求输入口令，但在“重置本靴（覆盖本靴）”时必须勾选确认框：
   - 我已确认将清空本靴数据（不可恢复）

## 非目标

1. 不在本次范围内重构 AI 模型策略或胜率算法。
2. 不新增复杂的上传格式（如 CSV 导入），仅支持 1/2/3 与格子点击录入。

## UX 设计

### 入口

- 位置：Dashboard 顶部右侧操作区（现有 DashboardHeader）
- 将“结束本靴”按钮替换为“上传数据”按钮
- 点击后路由跳转到 `/upload`（新页面）

### 上传数据页面（/upload）

**信息区**
- 展示当前靴号、当前已开局数、当前余额（只读）

**输入区（两种方式共用同一份数据源）**
- 珠盘路输入面板：6×12 固定格子
  - 点击循环：空→庄→闲→和→空
- 键盘输入：
  - 1=庄、2=闲、3=和
  - Backspace 删除最后一局
  - 连续输入追加多局（例如 112233…）

**局序列展示**
- 右侧列表：第 1 局…第 N 局
- 每行可删除该局（删除后局号重排为连续 1..N）

### 确认上传弹窗

**字段**
- 余额处理（单选）：保留当前余额（默认）/ 重置到 20000
- 提交动作（单选，默认：重置本靴）
  - 重置本靴（覆盖本靴）
  - 结束本靴（开启新靴）
- 仅当选择“结束本靴”时显示：
  - 是否执行深度学习（end_boot）（开关，默认开）

**摘要区（强制展示）**
- 当前靴号、当前已开局数、将写入的新局数 N
- 本次动作（覆盖本靴 / 新靴）
- 余额处理方式
- 若为覆盖本靴：列出将清空的数据范围

**防误触**
- 仅当选择“重置本靴”时：
  - 需要勾选“我已确认将清空本靴数据（不可恢复）”才允许提交

## 后端语义与接口

### 需求拆分

现有后端 `upload_games` 存在“续传校验”与“覆盖清场”混用的冲突：当 `is_new_boot=False` 时既要求从 next_game_number 接续，又会清空本靴数据。需要明确区分三种语义：

1. 追加本靴（Append）——不在本次 UI 提供
2. 覆盖本靴（Reset current boot）
3. 开启新靴（New boot）

### 建议接口参数（保持与现有兼容并扩展）

- POST `/api/games/upload`
  - `games`: 录入的 1..N 局结果
  - `mode`: `"reset_current_boot" | "new_boot"`
  - `run_deep_learning`: boolean（仅当 mode=new_boot 生效）
  - `balance_mode`: `"keep" | "reset_default"`

后端行为：
- `mode=reset_current_boot`：
  - 定位当前靴 boot_number（不变）
  - 删除本靴：GameRecord/BetRecord/MistakeBook/RoadMap/AIMemory
  - 重置内存态工作流（保持/重置余额由 balance_mode 决定）
  - 写入新 games，设置 next_game_number=N+1，进入分析流程
- `mode=new_boot`：
  - 若 run_deep_learning=true：先执行 end_boot 流程（深度学习），完成后进入等待新靴
  - boot_number+1，写入新 games
  - 余额处理同上

## 校验与边界条件

- games 长度 1..72
- 结果仅允许 庄/闲/和（键盘 1/2/3 映射后入库）
- 上传必须连续从第 1 局到第 N 局
- 若用户选择 reset_current_boot，必须在服务端再次确认当前状态允许重置（避免“深度学习中”时破坏状态）

## 测试验收

1. 进入 /upload，键盘输入 112233，格子与列表同步显示 6 局结果
2. 在格子上点击某一局循环切换，列表同步更新
3. 选择“重置本靴 + 保留余额”：
   - 提交前必须勾选确认框
   - 提交后本靴数据清空并重写，新局从 1..N 正确显示
4. 选择“结束本靴 + 执行深度学习 + 开启新靴上传”：
   - end_boot 执行后新靴号 +1，写入正确
5. 选择“结束本靴 + 不执行深度学习”：
   - 直接新靴写入并进入分析
