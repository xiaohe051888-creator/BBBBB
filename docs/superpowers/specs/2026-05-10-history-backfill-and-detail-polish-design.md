# History Backfill And Detail Polish Design

**Date:** 2026-05-10

**Goal:** 在已有“详情可读性重构 + reset_current_boot 历史快照保留”基础上，继续完成两项剩余工作：一是对已经被旧逻辑清空的历史局展示字段做自动回补；二是对 `推理详情` 做第二轮移动端优先精修，让它在手机端和长文阅读场景下更顺手、更易读。

**Architecture:** 后端新增“历史展示回补”服务，按 `boot_number + game_number` 聚合 `GameRecord / BetRecord / SystemLog` 证据链，对缺失展示字段进行可信恢复，并提供一个显式维护入口用于对现有历史数据执行一次回补；前端在保留现有抽屉结构的前提下，继续优化抽屉宽高、信息卡片、正文层级、滚动节奏和标签可读性，不改变业务流程。

**Tech Stack:** React, TypeScript, Ant Design Drawer/Tag/Collapse, FastAPI, SQLAlchemy, pytest, Vitest

---

## 1. Problem Statement

### 1.1 已修复但仍未彻底完成的历史问题

上一轮修复已经阻止了 `reset_current_boot` 继续清空历史展示字段，但在修复上线之前被错误清空的老历史局仍然存在：

- `predict_direction = null`
- `predict_correct = null`
- `profit_loss = 0.00`
- `settlement_status` 缺失或退化

这些局并非真的“从未有过预测和结算”，而是历史展示值已被旧逻辑破坏。当前数据库里还保留着一部分恢复线索：

- 有些局保留了 `BetRecord`
- 有些局保留了开奖与结算日志
- 有些局保留了 AI 分析日志

因此目标不是“猜数据”，而是“只用现有证据自动恢复能确认的字段”。

### 1.2 详情页仍然存在的第二层体验问题

上一轮已把详情页改成正确顺序，但仍有以下问题：

- 手机端抽屉阅读高度与滚动节奏还不够友好
- 结论卡、五路卡片和来源说明的视觉强弱还可以更明确
- 正文虽然可读了，但长段落仍然缺少足够舒适的阅读密度
- 方向、把握度、来源标签可读性仍有提升空间
- 技术说明折叠区与主内容之间的“分离感”还不够强

这意味着它已经“能看清”，但还没有达到“看着舒服、扫读顺手”。

## 2. Backfill Design

### 2.1 回补目标

只回补以下“用户已感知的历史展示字段”：

- `prediction_mode`
- `predict_direction`
- `predict_correct`
- `settlement_status`
- `profit_loss`
- `balance_after`

不回补以下运行态字段：

- `pending_bet_*`
- `analysis_outcome`
- `predict_confidence`（若没有可靠证据则不补）
- 当前内存会话状态

### 2.2 证据优先级

回补按以下优先级进行：

1. `BetRecord`
2. `SystemLog`
3. `GameRecord` 现存字段

理由：

- `BetRecord` 对“下注方向 / 下注金额 / 是否已结算”最可信
- `SystemLog` 对“AI预测方向 / 开奖结果 / 盈亏 / 余额变化”有补充价值
- `GameRecord` 现存字段只能作为最后兜底，避免覆盖真实证据

### 2.3 可恢复字段规则

#### `predict_direction`

满足以下任一条件即可恢复：

- 存在该局 `BetRecord.direction`
- 或日志中存在 `AI对第X局推理完成：预测【庄/闲】`

若两种来源冲突：

- 优先 `BetRecord.direction`
- 冲突时记录 `warning` 日志，不写回冲突来源值

#### `predict_correct`

需要同时满足：

- 可确定 `predict_direction`
- 可确定该局 `GameRecord.result`

恢复公式：

- `和局` 时不强行写 `True/False`，保持现有规则语义
- `庄/闲` 时直接比较方向与开奖结果

#### `profit_loss`

满足以下任一条件才恢复：

- `BetRecord.profit_loss` 或 `settlement_amount` 可推导
- 日志中存在明确的 `盈亏+9.50 / 盈亏-10.00 / 盈亏+0.00`

没有明确证据时绝不写 `0.00`。

#### `settlement_status`

若存在已结算注单或开奖/结算日志，则恢复为 `已结算`。  
若该局明显处于未开奖态，则不改。

#### `balance_after`

仅在结算日志或注单结算记录中存在可信余额时恢复。  
没有余额证据时宁可留空。

#### `prediction_mode`

若该局日志存在 `单AI`、`规则兜底` 或现有记录中已有模式，则恢复；否则保持空。

### 2.4 实现位置

新增后端服务文件：

- `backend/app/services/game/history_backfill.py`

核心职责：

- 扫描指定靴号的 `GameRecord`
- 识别历史展示缺失的局
- 查询对应 `BetRecord` 与 `SystemLog`
- 生成结构化回补结果
- 将可确认字段写回数据库
- 返回“扫描多少局、修复多少局、跳过多少局、冲突多少局”的摘要

### 2.5 触发方式

新增一个显式维护入口，避免偷偷在普通流程里回填旧数据：

- 推荐新增系统维护接口，例如：
  - `POST /api/system/backfill-history`

请求参数支持：

- `boot_number`：可选，默认当前靴
- `limit_games`：可选，仅针对最近若干局
- `dry_run`：可选，先看将要修什么，不落库

这样可以支持：

- 本次我在线上亲自执行一次修复
- 后续如果还发现旧历史局缺失，可再次手动执行

### 2.6 安全原则

回补必须满足：

- 只补空字段或明显错误的占位字段
- 不覆盖已有明确正确值
- 不猜测没有证据的数据
- 冲突时保守跳过并记日志

## 3. Detail Polish Design

### 3.1 保持结构不变

保留现有顺序：

1. `本局结论`
2. `最终为什么押这个方向`
3. `五条路怎么看`
4. `来源说明`
5. `技术说明`

这次只做“第二轮阅读体验精修”，不再改信息架构。

### 3.2 具体精修项

#### 抽屉层

- 抽屉 body 增加更稳定的底部留白
- 移动端下限制内容最大宽度并优化内边距
- 避免最后一个 section 贴底或被遮挡

#### 结论卡

- 方向文案更突出
- 把握度与来源标签的对比度和间距更清楚
- `short_reason` 改成更舒服的阅读块

#### 五条路卡片

- 标题区和正文区再拉开层级
- 标签与正文之间增加呼吸感
- 长文本段落进一步增加行高
- 移动端保持单列，但减少横向挤压感

#### 来源说明与技术说明

- 来源说明继续保留，但颜色与视觉权重低于结论区
- 技术说明折叠区用更明显的“附加信息”样式，不和主内容争主次

### 3.3 不做的事情

- 不改成新路由页
- 不换成全屏 modal
- 不重做首页分析卡
- 不改后端 `analysis_outcome` 契约

## 4. Testing

### 4.1 后端测试

新增测试覆盖：

- 有 `BetRecord` + `GameRecord` 的局，回补成功
- 只有 `SystemLog` 的局，能恢复可确认字段
- 证据冲突的局，不应误写
- `dry_run` 不落库，只返回计划修复摘要

### 4.2 前端测试

更新 `AnalysisDetailDrawer` 测试，至少覆盖：

- 关键区块顺序仍正确
- 结论卡、五路卡片、来源说明、技术说明仍存在
- 第二轮精修后关键标签与文本依然可见

### 4.3 真实复测

实现完成后需要：

1. 执行一次线上历史回补
2. 刷新页面确认老历史局不再大面积 `- / 0.00`
3. 打开详情抽屉并在手机宽度下复测阅读体验

---

## 5. Scope Boundaries

本次只做：

- 旧历史局自动回补
- 详情页第二轮精修

本次不做：

- 所有历史数据的全量重建平台
- 下注规则或开奖流程改动
- 首页大结构重做

---

## 6. Success Criteria

满足以下条件视为完成：

- 已被旧逻辑清空的历史局中，能自动恢复的字段被尽量恢复
- 没有证据的字段不会被乱补
- 页面历史表格里不再长期大面积出现错误的 `- / 0.00`
- 详情页在手机端阅读时明显更舒服，结论与五路卡片更易扫读
