# 单AI解析失败工作流修复设计

## 背景

当前单AI模式已经固定为 DeepSeek `V4 Pro + thinking=enabled`，但线上仍出现一种严重错位现象：

- 模型返回内容不可解析
- 后端没有把这次分析视为失败
- 系统反而生成默认结果（例如 `庄 / 0% / 解析失败`）
- 前端继续显示“已完成判断”
- 详情页展示“本局建议：庄”，但理由、最终原因却是“解析失败”

这会给用户造成错误认知：看起来像模型真的完成了分析，其实只是系统把一次无效输出伪装成了成功结果。

## 问题定义

本次需要修复的不是单点文案，而是整个单AI结果工作流的真实性。

当前错误链路是：

1. 单AI调用成功返回文本
2. 返回文本不满足严格 JSON
3. 解析器返回默认结构：
   - `final_prediction = 庄`
   - `confidence = 0.0`
   - `summary = 解析失败`
4. 后端把这份默认结构继续当成合法 `analysis_outcome`
5. 状态流转到“分析完成”
6. 前端主卡和详情抽屉按成功态展示

本次修复目标：

1. 解析失败不能再伪装成成功分析结果
2. 单AI只有在拿到有效结构化结果时，才允许进入“分析完成”
3. 单AI解析失败时，要么进入规则兜底，要么停留在明确失败态
4. 前端展示必须和后端真实状态一致
5. 同时提升 DeepSeek 正式输出可解析率，减少再次触发解析失败

## 方案对比

### 方案 A：只改前端展示

做法：

- 把“解析失败”换成更柔和的中文
- 隐藏 `0%` 或默认方向

优点：

- 改动小

缺点：

- 后端仍然在制造假成功
- 只是遮丑，不解决根因

结论：

不采用。

### 方案 B：只在解析失败时抛异常

做法：

- 解析器不再返回默认结果
- 直接抛出异常

优点：

- 能阻止伪成功

缺点：

- 如果不一起改状态机、日志和前端，会产生新错位
- 用户可能看到更混乱的状态残留

结论：

不单独采用。

### 方案 C：全链路重构

做法：

- 解析器严格区分“有效结果”和“无效结果”
- 单AI运行链只接受有效结果
- 无效结果统一进入规则兜底或失败态
- 统一日志事件与前端展示
- 重构默认提示词和解析器，尽量减少不可解析输出

优点：

- 真正修复工作流
- 状态、日志、界面三方一致
- 便于后续定位问题

结论：

采用方案 C。

## 设计

### 1. 结果真实性原则

新增明确原则：

- 只有当单AI返回的结构满足最小有效条件时，才能认定为“分析完成”
- 解析失败、字段缺失、方向无效、置信度非法，都属于“无效结果”

最小有效条件：

- `final_prediction` 必须为 `庄` 或 `闲`
- `confidence` 必须能转成 `0 ~ 1` 的数字
- `summary` 必须为非空文本
- `reasoning_detail` 必须为非空文本

不满足上述条件时，不再生成默认成功结果。

### 2. 单AI解析器重构

当前 `_parse_model_json()` 的问题是：

- 解析失败时返回“默认庄 + 0% + 解析失败”

本次改成：

- 解析失败时抛出明确异常，例如 `SingleAIParseError`
- 异常中携带：
  - 原始响应摘要
  - 失败类型
  - 失败原因

失败类型至少区分：

- 非 JSON 文本
- JSON 截断
- 缺少必须字段
- 字段值非法

这样可以让上层明确知道“这是一次无效结果”，而不是“低把握的成功结果”。

### 3. 单AI工作流重构

在 `run_ai_analysis()` / follow-up analysis 链路中，单AI分成两段：

1. **生成阶段**
   - 调用 DeepSeek
   - 解析为结构化结果
2. **验收阶段**
   - 校验结果是否满足最小有效条件
   - 只有验收通过才写入 `analysis_outcome`

如果生成或验收失败：

- 不写入单AI成功态预测结果
- 不写入“已完成判断”快照
- 不记 `LOG-MDL-001` 这类成功日志
- 进入规则兜底分支

如果规则兜底成功：

- 由规则兜底产出真实 `analysis_outcome`
- 前端展示“备用判断”
- 日志记为“自动改用备用判断”

如果规则兜底也失败：

- 才进入明确失败态
- 前端不再显示建议方向

### 4. 状态机约束

新增状态约束：

- 单AI解析失败时，不能进入 `showCompletedAnalysis = true` 的可展示成功态
- `waiting_reveal` 必须建立在“已形成有效预测并完成下注”的前提上
- 任何仅包含错误文案、没有有效预测结果的状态，都不得显示“本局建议”

换句话说：

- “分析完成”必须对应真实有效结果
- “备用判断”必须对应真实有效的规则结果
- “解析失败”只能作为诊断，不允许出现在成功结果主文案里

### 5. 前端展示收口

前端主卡 `AnalysisPanel` 与详情抽屉 `AnalysisDetailDrawer` 的展示原则同步调整：

- 如果 `analysis_outcome` 是有效结果：
  - 正常显示建议方向、置信度、原因、详情
- 如果当前是规则兜底结果：
  - 显示备用判断相关中文说明
- 如果后端没有有效结果：
  - 不显示建议方向
  - 不显示“已完成判断”
  - 只显示“系统正在重新判断”或“本次未生成有效判断”

前端不再兜底把旧字段强行拼成成功态结果，尤其不能把错误信息包装成建议方向。

### 6. 日志与诊断重构

当前日志层面存在“解析失败但仍记成功完成”的风险，本次统一：

- 单AI有效完成：
  - 记成功日志
- 单AI解析失败后规则兜底成功：
  - 记“智能分析无有效结果，系统改用备用判断”
- 单AI解析失败且规则兜底失败：
  - 记明确失败日志

诊断信息要能回答三个问题：

1. 上游有没有返回内容
2. 返回内容为什么无效
3. 系统最终是成功兜底了，还是彻底失败了

### 7. 单AI提示词与解析成功率

虽然工作流必须在失败时说真话，但还需要同时减少失败概率。

因此本次继续强化单AI默认提示词：

- 明确允许内部思考，但最终只输出 JSON
- 禁止 Markdown
- 禁止额外解释
- 明确字段名称、字段类型和取值范围
- 在正式模板中减少无关噪声

同时保留当前固定约束：

- 单AI固定 DeepSeek `V4 Pro`
- 固定开启深度思考
- 官方地址固定为 `https://api.deepseek.com`

## 测试策略

本次至少补齐以下回归：

1. 单AI返回非 JSON 时，不再产出默认成功结果
2. 单AI返回缺字段 JSON 时，不再产出默认成功结果
3. 单AI解析失败时，`run_ai_analysis()` 应进入规则兜底
4. 单AI解析失败时，不记录单AI成功分析日志
5. 规则兜底成功时，前端显示备用判断，而不是“解析失败 + 本局建议”
6. 无有效结果时，前端主卡不显示建议方向和“已完成判断”
7. 详情抽屉不再展示“解析失败”作为成功理由

## 风险与边界

- 本次不移除规则兜底，而是让它在单AI无效结果时承担正确职责。
- 本次不重做三模型协作链路，只处理单AI与其相关展示。
- 如果线上数据库里已经保存了旧版“解析失败伪成功”历史记录，本次只修新工作流，不批量回写旧脏数据；旧数据如需修复应单独做历史清洗。
